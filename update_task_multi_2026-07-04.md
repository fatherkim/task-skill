# Отчёт: task-multi + апгрейд конвейера /task (2026-07-04)

Полный блок работ одной сессии: от session-lock до мультиоркестраторного режима, анализа конкурентов
с GitHub и порта 10 фич. Коммиты test_kvdb: 9ae9523 → aca6bd8 → e08437f → f8b7171/cf1e3e9 → ddb0a9f → 886535a → 9c5097f.

## 1. Session-lock одиночного режима (9ae9523)

Предохранитель от второго оркестратора: файл `_LOCK` в трекере, атомарный захват (O_EXCL),
owner-метка, маркер протухания >24ч, `--force`-перехват только с подтверждения пользователя,
напоминания в мутирующих командах. Команды `lock`/`unlock`.

## 2. Мультиоркестраторный режим — скилл /task-multi (aca6bd8)

Вариант B (выбор пользователя): каждый оркестратор в своём git worktree.

- **Источник правды трекера** — служебная ветка `task-sync` (только файлы трекера, без кода);
  рабочие копии в worktree — кэши.
- **Транзакции**: каждая мутация = глобальный flock-mutex (в общем `.git`, действует на все
  worktree) → sync(pull) → операция → sync(push). Sync — 3-way merge деревьев (base из
  `.sync-base`, ours-снапшот через временный индекс, theirs = tip ветки) без checkout.
- **Владение**: `start N --owner` — claim; чужая in_progress — отказ.
- **`merge-main`** — слияние ветки задачи в main под mutex: через чистый main-worktree или
  `merge-tree --write-tree` вообще без checkout.
- CLI общий с /task (суперсет, обратно совместимый); скилл `/task-multi` — надстройка-протокол
  над `/task` без дубля кода.
- Пойманные при тестировании баги: `ls-tree` без `--full-tree` фильтровал по префиксу cwd
  (sync молча «актуален»), untracked-файлы ложно блокировали merge-main, `multi off` смотрел
  несинхронизированный трекер.

## 3. Фиксы коллизий multi-режима (e08437f)

По результатам адверсариального самоанализа («какие коллизии остались»):

1. `close`/`block`/`unblock` — owner-guard: `--owner` обязателен, сверка владельца И worktree;
   чужая задача — отказ, `--force` только с подтверждения.
2. Детект пропущенного `sync --adopt`: `_MULTI` в ветке task-sync есть, локально нет → отказ
   на первой операции.
3. Дубль метки owner: `start` фиксирует worktree сессии; та же метка из другого worktree — отказ.
4. Атомарный `new`: задача рождается коммитом в task-sync (blob→tree→commit→update-ref до
   локальной записи) — crash-окно дубля id закрыто.
5. `list`/`view` подтягивают свежий трекер; `calibrate` обёрнут в транзакцию.
6. `sync` подсвечивает дубли id; `merge-main` предупреждает об уехавшем main.

Нефиксируемое механикой (зафиксировано в SKILL.md процессно): семантический конфликт при чистом
git-merge, ручной `git merge` мимо merge-main, логические ресурсы поверх непересекающихся files.
