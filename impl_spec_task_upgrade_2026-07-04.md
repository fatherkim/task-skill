# Implementation Spec: апгрейд скиллов /task и /task-multi (порт 10 фич из анализа конкурентов)

Источник требований: docs/analysis_task_competitors_2026-07-04.md (таблица «Что переносить», колонка «Скилл»).
Дата: 2026-07-04. Статус: SPEC — к исполнению.

## 0. Рамки и инварианты

- Правится ТОЛЬКО канонический CLI `.claude/skills/task/scripts/task.py` + `.claude/skills/task/SKILL.md` + `.claude/skills/task-multi/SKILL.md` + тестовый стенд в scratchpad.
- `agent_tasks/` НЕ трогать до финального шага деплоя (Ш10). `agent_tasks/_cli.py` обновляется копированием каноника ПОСЛЕ прохождения всех тестов (0047 закрыта, lock свободен — деплой разрешён).
- Python stdlib only. Никаких зависимостей.
- Одиночный режим (без `_MULTI`): поведение существующих команд НЕ меняется, кроме явно описанного (новые опции — опциональны, новые предупреждения — только при реальных находках). Регрессия single-mode — обязательный тест.
- Все сообщения CLI — по-русски, в стиле существующих.
- После каждой фичи — `python3 -m py_compile`. Тесты — в конце (Ш9), одним прогоном.
- Git-коммиты по ходу НЕ делать: коммитит оркестратор после ревью.

## Ш1. Schema-version gate (фича #7, multi)

CLI:
1. Константа `SCHEMA_VERSION = 2` (v1 = формат до owner/worktree; v2 = текущий+Runs).
2. Файл `_SCHEMA` в каталоге трекера, содержимое — одно число + перевод строки. Файл СИНХРОНИЗИРУЕТСЯ в multi (НЕ добавлять в SYNC_EXCLUDE).
3. Хелпер `_schema_check(d)`: файла нет → OK (legacy v1); число > SCHEMA_VERSION → `sys.exit("ОТКАЗ: трекер схемы vN, CLI понимает только v%d — обнови tasks/_cli.py из каноника")`; нечисловое содержимое → та же ошибка с пометкой «битый _SCHEMA».
4. Вызов `_schema_check` в начале всех мутирующих команд (new, _set_status, close, archive, calibrate --set) и в `_sync`.
5. `cmd_init` и `multi on` записывают `_SCHEMA` = SCHEMA_VERSION (init — всегда; multi on — если файла нет или число меньше).

## Ш2. Run-лог попыток (фича #4, оба) + команда return (фича #3, оба)

CLI:
1. Хелпер `_append_run(body, outcome, note="", spent="")`: находит секцию `## Runs` в теле (если нет — добавляет в конец `\n## Runs\n`), дописывает строку `- {ISO-датавремя} — {outcome}{, owner=X если есть}{, spent=...}{ — note}`. Возвращает новое тело.
2. Новая команда `return N --reason "..." [--owner X] [--spent "..."]` (cmd_return):
   - multi-транзакция (`_multi_begin`/`_multi_end`) + `lock_notice` + `_schema_check`;
   - guard: задача должна быть `in_progress`; `_owner_guard(d, meta, owner, force, "return")`;
   - инкремент поля frontmatter `returns:` (int, default 0);
   - `_append_run(body, "returned", note=reason, spent=...)` + append-секция `## Возврат N (дата)\n{reason}` (как у block);
   - статус остаётся `in_progress` (задача у того же исполнителя);
   - **circuit breaker**: если новое `returns >= 3` — печать `⚠ ЦИКЛ: 3 возврата — СТОП, эскалируй пользователю (правило circuit breaker в SKILL.md)`; сам возврат при этом выполняется (мягкий предохранитель, не блок).
3. `close`: перед записью — `_append_run(body, "closed", note=<первые 80 символов note>, spent=args.spent)`.
4. `block`: `_append_run(body, "blocked", note=reason)`.
5. `write_task`: ключ `returns` в списке frontmatter-ключей (после `worktree`).
6. Парсер: `return` с аргументами id, --reason (обязателен), --owner, --force, --spent; dispatch.

## Ш3. ready --waves + детект циклов (фича #2, оба)

CLI:
1. `ready --waves`: топологическая раскладка Кана всех open-задач:
   - узлы = open; done/archived deps считаются выполненными; deps на blocked/in_progress держат задачу вне волн (она попадёт в волны после их закрытия — в выводе секция «ждут: NNNN (блокирована 000X)»);
   - волна k = open-задачи, все deps которых done ИЛИ в волнах <k;
   - внутри волны проверить пересечения `files` попарно: пересекающиеся пометить `!` с указанием пар (как в текущем ready) и вынести конфликтующую (бОльший id) в следующую волну — волна должна быть безопасна для одновременного диспатча целиком;
   - вывод: `Волна 1: 0005, 0007`, `Волна 2: 0009 (после 0005)`, затем «ждут»-секция.
2. Детект циклов: если Кан застрял (остались узлы с невыполнимыми deps, не объяснимые blocked/in_progress/отсутствующими задачами) — `⚠ ЦИКЛ зависимостей: 0005 -> 0007 -> 0005` (вывести участников), exit 1 в `--waves`; в обычном `ready` — только предупреждение, exit 0.
3. Обычный `ready` без флага — поведение не меняется (плюс предупреждение о цикле, если есть).
4. Валидация deps на `new --depends`: несуществующий id (ни живой, ни архивный) → предупреждение (не отказ — задача-зависимость может создаваться следом).

## Ш4. Risk-score (фича #5, оба)

CLI:
1. `new --risk {0,1,2,3}` → frontmatter `risk:`; без флага поле не пишется.
2. `write_task`: ключ `risk` (после `title`).
3. `list`/`_INDEX.md`: колонка Risk НЕ добавляется (не раздувать таблицу); `view` и так покажет frontmatter. `ready`/`ready --waves`: суффикс ` [r3]` у задач с risk>=2.
4. Шаблон TEMPLATE не меняется.

## Ш5. merge-main: авто-fallback в integration + --resolve worktree (фича #1, multi)

CLI (cmd_merge_main):
1. Случай «base checked out в ГРЯЗНОМ дереве» (сейчас — отказ): вместо отказа — авто-fallback:
   - целевая ветка fallback: `integration` (создать от текущего tip base, если нет: `git branch integration <base>`);
   - рекурсивно выполнить ту же merge-логику с base=integration (merge-tree / чистый worktree / update-ref);
   - печать: `main занят (грязное дерево X) — влито в integration. Финализация: git merge --ff-only integration (в дереве main), затем ветку integration можно удалить.`;
   - флаг `--no-fallback` возвращает старое поведение (отказ) — для случаев, когда пользователь хочет сам почистить дерево.
2. Новый флаг `merge-main N --resolve`: для случая, когда `merge-tree` вернул КОНФЛИКТ:
   - создать временный worktree: `git worktree add <repo-родитель>/<repo>-merge-NNNN -b _merge/NNNN <base>` (детач не нужен — временная ветка);
   - в нём запустить `git merge --no-ff <task-branch>` (ожидаемо упадёт с конфликтами);
   - напечатать путь worktree, список конфликтующих файлов (`git diff --name-only --diff-filter=U`) и инструкцию: «разрули конфликты, git add + git commit, затем: merge-main N --branch _merge/NNNN; после успеха: git worktree remove <путь> и git branch -D _merge/NNNN»;
   - сам merge-main при этом завершает работу exit 0 (он подготовил площадку).
   - если worktree/ветка `_merge/NNNN` уже существуют — переиспользовать с предупреждением.
3. Без `--resolve` конфликтный случай — как сейчас (отказ с подсказкой), но в подсказку добавить упоминание `--resolve`.

## Ш6. Depends-замержены-гейт в merge-main (фича #8, multi)

CLI (cmd_merge_main, до merge-tree):
1. Прочитать задачу N (если id указан): для каждого dep из `depends`:
   - dep-задача не найдена → предупреждение;
   - статус dep != done → ОТКАЗ: `зависимость 000D не закрыта — сначала заверши и влей её` (обход `--force`);
   - статус done, ветка `task/000D` существует и НЕ ancestor base → ОТКАЗ: `зависимость 000D закрыта, но ветка task/000D не влита в {base}` (обход `--force`);
   - ветки нет (уже удалена после мержа) → OK.
2. При `--branch` без id — гейт пропускается (нет спеки).

## Ш7. Дубль-ID fix-промпт (фича #10, multi)

CLI: `_warn_dup_ids` — после строки `⚠ ДУБЛЬ id ...` печатать готовый блок:
```
Задание на разбор дубля (отдай свободному оркестратору/исполнителю):
  1. Определи, какой файл настоящий NNNN (git log ветки task-sync: кто первый).
  2. Второму файлу выдай следующий свободный id: mv + правь id: в frontmatter.
  3. grep по 'NNNN' в tasks/*.md — поправь depends, ссылающиеся на перенумерованный.
  4. python3 tasks/_cli.py index && python3 tasks/_cli.py sync
```
(текст с подставленными реальными id и именами файлов).

## Ш8. SKILL.md обоих скиллов (фичи #3, #5, #6, #9 процессные части)

`.claude/skills/task/SKILL.md`:
1. Таблица CLI: строки для `return N --reason`, `ready --waves`, `new --risk`; упоминание `## Runs` и `returns:`.
2. Раздел «Приёмка»: **правило circuit breaker** — 3-й возврат одной задачи = стоп-сигнал: не возвращать в 4-й раз, а эскалировать пользователю (варианты: переспецификация, смена исполнителя/модели, разбиение задачи); CLI напоминает при returns>=3.
3. Раздел «Приёмка»: **правило дедупа фидбека** — перед повторным возвратом сравни новый провал с текстом последнего `## Возврат N`: тот же симптом и дифф не изменился → это не новая итерация, а зацикливание — эскалируй, не повторяй возврат (счётчик всё равно инкрементируй через `return`).
4. Раздел «Декомпозиция»: **правило risk-score** — при `new` проставляй `--risk`: 0 = тривиальное (docs/тесты/конфиг без логики), 1 = локальная логика, 2 = общие модули/данные, 3 = flash/загрузчик/протокол/безопасность. Маппинг на приёмку: risk 0 — verify без ревьюера; 1 — ревью диффа читающим субагентом; 2 — полный конверт ревьюера-опровергателя; 3 — ревьюер + обязательный HW/e2e прогон.
5. Новый подраздел «Уровни автономии»: `автономия: interactive|supervised|autonomous` в вызове скилла; interactive (дефолт) — AskUser на все force/деструктивные и на каждый circuit-breaker; supervised — force только с вопросом, остальное молча с отчётом; autonomous — вопросов нет, все force-операции ЗАПРЕЩЕНЫ (вместо них — блокировка задачи + отчёт), отчёт в конце. Указание: уровень действует до конца сессии.
6. Правка строки про multi-команды: добавить `return`, `ready --waves`.

`.claude/skills/task-multi/SKILL.md`:
1. Таблица CLI-добавок: `merge-main` — новые поведения (auto-fallback integration, `--resolve`, depends-гейт, `--no-fallback`); `_SCHEMA`-гейт; дубль-ID fix-промпт.
2. Протокол, шаг «Приёмка»: конфликт → сначала `--resolve`-worktree (сам или исполнителем), исполнителю ребейз — вторая опция; финализация integration→main в шаге «Финализация».
3. Раздел «Правила»: schema-gate (что делать при отказе — обновить _cli.py из каноника); автономия — ссылка на раздел /task + уточнение: в autonomous `multi off --force` и `unlock --force` запрещены.
4. Ограничения: пункт про уехавший main дополнить: предупреждение merge-main теперь печатается всегда, при risk>=2 задач в ветке — перетест обязателен (процессно).

## Ш9. Тесты (scratchpad)

Расширить `multitest.sh` (или рядом `multitest2.sh`), добавив сценарии:
- T1: `_SCHEMA` пишется init/multi on; ручная запись `99` → мутирующая команда отказывает; missing → работает.
- T2: `return 2 --reason x --owner B` из B: returns=1, секция Возврат, строка в Runs; чужой return из A → отказ; 3 возврата → появляется ⚠ ЦИКЛ.
- T3: close/block пишут строки в `## Runs`.
- T4: `ready --waves`: 4 задачи (0002 depends 0001; 0003 независима; 0004 files пересекается с 0003) → Волна 1: 0001, 0003; Волна 2: 0002, 0004 (0004 отложена из-за файлов); цикл A->B->A → exit 1 с перечислением.
- T5: `new --risk 3` → risk в frontmatter; `ready` показывает [r3].
- T6: merge-main при грязном main → авто-fallback: ветка integration существует, merge попал туда, main не тронут; `--no-fallback` → старый отказ.
- T7: merge-main конфликт + `--resolve` → создан worktree `_merge/NNNN`, в нём конфликтные маркеры; после ручного резолва (в тесте: git checkout --theirs + commit) `merge-main N --branch _merge/NNNN` проходит.
- T8: depends-гейт: задача 0002 depends 0001, 0001 не done → merge-main 2 отказ; закрыли и влили 0001 → проходит.
- T9: дубль id → в выводе sync есть блок «Задание на разбор дубля» с реальными id.
- T10: single-mode полная регрессия (как раньше) + `return` в single (без owner-требований) + `ready --waves` в single.
Все прежние 17 проверок должны остаться зелёными (обнови ожидания там, где вывод дополнился новыми строками).

## Ш10. Деплой (после зелёных тестов и ревью)

1. `diff` каноника и `agent_tasks/_cli.py` (ожидаемо большой), затем `cp .claude/skills/task/scripts/task.py agent_tasks/_cli.py`.
2. В живом трекере `_SCHEMA` НЕ создавать руками (появится при следующем multi on; single работает без него).
3. Прогнать в live-репо read-only команды: `list`, `ready`, `ready --waves`, `stats` — убедиться, что живой трекер читается.
