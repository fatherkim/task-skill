# task-multi — несколько оркестраторов на одном репозитории

Надстройка над скиллом `/task` (вариант B: worktree на оркестратора). **Все правила базового скилла действуют без изменений** — декомпозиция, спеки, конверты исполнителя/ревьюера, приёмка, vacuous-pass контроль, учёт токенов. Прочитай `${CLAUDE_SKILL_DIR}/../task/SKILL.md`, если его нет в контексте. Здесь — только то, что добавляет multi-режим.

## Когда НЕ использовать

- **Малый проект / короткая задача.** Один оркестратор и так справляется — mutex,
  owner-guard и `task-sync` ветка это накладные расходы без выгоды, если параллелить
  нечего.
- **Один оркестратор достаточен.** Multi решает гонку ЗА трекер между несколькими
  *оркестраторами*; параллельные исполнители одного оркестратора уже покрыты
  обычным `/task` (worktree на задачу, раздел «Режим исполнения»).
- **Среда без git worktree или без POSIX flock.** Модель держится на них (mutex —
  `flock` в общем `.git`, изоляция — `git worktree`); без обоих — используй только
  базовый `/task`.
- **Нет реальной параллельности задач.** Если `ready --waves` в базовом режиме и
  так показывает одну волну — второй оркестратор её не ускорит, только добавит
  координационные накладные расходы.

CLI общий с `/task`: `python3 <tracker>/_cli.py` (канонический источник — `.claude/skills/task/scripts/task.py`). Multi-команды уже в нём; отдельного CLI у этого скилла нет — не плоди копий.

## Модель

- Источник правды трекера — служебная ветка **`task-sync`** (только файлы каталога трекера, без кода). Рабочие копии трекера в каждом worktree — кэши.
- Каждая мутирующая команда CLI в multi-режиме — транзакция: глобальный **mutex** (flock в общем `.git`, действует на все worktree) → sync(pull) → операция → sync(push). `ready`/`list`/`view`/`multi off` тоже синхронизируются перед показом/решением — устаревшую картину увидеть нельзя.
- Конфликтов по файлам задач не бывает при дисциплине владения: задачу мутирует только оркестратор, захвативший её через `start N --owner`.
- `_INDEX.md`, `_LOCK`, `.sync-base`, `_cli.py` не синхронизируются (индекс регенерируется локально). `_MULTI` синхронизируется — флаг режима общий.
- Эксклюзивный `lock` из `/task` в multi-режиме **запрещён** (CLI откажет) — взаимоисключимые предохранители.

## CLI-добавки

| Команда | Что делает |
|---|---|
| `multi on` | включить режим: создаёт `_MULTI` + ветку task-sync; откажет при живом `_LOCK` |
| `multi off [--force]` | выключить; откажет при in_progress задачах (--force — только с подтверждения пользователя) |
| `multi status` | режим, ветка, in_progress с owner'ами, worktrees |
| `sync [--adopt] [--prefer ours\|theirs]` | ручная синхронизация; `--adopt` обязателен в свежем worktree (заменяет локальную копию трекера состоянием ветки); `--prefer` — только после ручного разбора конфликта |
| `start N --owner <метка>` | claim задачи: `--owner` в multi-режиме обязателен; чужая in_progress — отказ; фиксирует worktree сессии (ловит две сессии с одной меткой) |
| `close/block/unblock N --owner <метка> [--force]` | в multi-режиме — только владельцем задачи (owner + worktree сверяются); чужая — отказ, `--force` только с подтверждения пользователя |
| `merge-main N [--branch X] [--base main] [--resolve] [--no-fallback] [--force]` | влить `task/NNNN` в main под глобальным mutex; без checkout, если main нигде не выложен |
| `merge-main` — грязный main | **авто-fallback в `integration`**: ветка `integration` создаётся от tip main (если нет), merge уходит туда, main не тронут; печатается финализация `git merge --ff-only integration`. `--no-fallback` возвращает старое поведение (отказ) |
| `merge-main N --resolve` | при КОНФЛИКТЕ merge-tree готовит временный worktree `_merge/NNNN` (ветка от base): разрули руками → `git add`+`commit` → `merge-main N --branch _merge/NNNN` → `git worktree remove` + `git branch -D`. Уже существующий worktree/ветка — переиспользуются с предупреждением |
| `merge-main N` — depends-гейт | до merge проверяет: каждый dep задачи N должен быть `done` И его ветка `task/000D` влита в base (иначе отказ). Обход — `--force` (только с подтверждения пользователя). При `--branch` без id гейт пропускается |
| `_SCHEMA`-гейт | файл `_SCHEMA` (версия схемы, синхронизируется) читается перед каждой мутацией: версия новее, чем понимает CLI → отказ «обнови `_cli.py` из каноника». `multi on`/`init` пишут актуальную версию |
| дубль-ID fix-промпт | `sync` при `⚠ ДУБЛЬ id` печатает готовое «Задание на разбор дубля» (кто первый по git-логу task-sync, перенумеровать второй, поправить depends, index+sync) |

## Протокол оркестратора

0. **Вход.** Метка owner = инструмент+дата+время+**случайный хвост** (напр. `cc-0705-1030-x7k2`) — две сессии, запущенные в одну минуту, не должны совпасть меткой (CLI дополнительно ловит совпадение по worktree, но метку делай уникальной сам).
   ```bash
   python3 <tracker>/_cli.py multi status        # режим включён? кто уже работает?
   # если режим ещё не включён (по согласованию с пользователем): multi on
   git worktree add ../<repo>-orch-<owner> -b orch/<owner> main
   cd ../<repo>-orch-<owner> && python3 <tracker>/_cli.py sync --adopt
   ```
   Дальше вся сессия — из этого worktree. В основном дереве репозитория оркестратору делать нечего.
1. **Декомпозиция/планирование** — как в `/task`. `new` и `ready` сами синхронизируются: `ready` показывает актуальные чужие claim'ы. Пересечения `files` между оркестраторами ловятся так же, как между задачами.
2. **Взятие задачи** — только `start N --owner <owner>`. Отказ = задачу уже держит другой оркестратор: выбери другую, не пересекающуюся по `files`.
3. **Диспатч исполнителей** — как в `/task`, worktree исполнителя ответвляется от worktree оркестратора:
   ```bash
   git worktree add ../<repo>-wt-NNNN -b task/NNNN orch/<owner>
   ```
   Спеки исполнителю видны без коммита в код-ветку: конверт даёт АБСОЛЮТНЫЙ путь к файлу спеки в дереве оркестратора (трекер живёт на task-sync, в код-ветки его в multi-режиме не коммитят до самого конца).
4. **Приёмка** — как в `/task` (`verify N --base main`), но слияние — `merge-main N`, не ручной `git merge`:
   - конфликт merge-tree → **сначала `merge-main N --resolve`**: CLI готовит worktree `_merge/NNNN`, разрули конфликт (сам или отдай исполнителю с контекстом), закоммить, затем `merge-main N --branch _merge/NNNN`. Вторая опция — ребейз исполнителем: смержить main в `task/NNNN` в его worktree и повторить `merge-main`;
   - main выложен в грязном дереве (обычно основное дерево пользователя) → merge-main сам уходит в `integration` (авто-fallback) и печатает финализацию; если хочешь отказ вместо fallback — `--no-fallback`;
   - depends-гейт: merge-main не даст влить задачу, пока её зависимости не закрыты и не влиты в base — сначала прими и влей их;
   - коммит «task NNNN: приёмка» для трекера НЕ нужен — состояние уехало в task-sync автоматически.
5. **Выход.** Закрой/верни свои задачи, `git worktree remove` исполнительских деревьев, сообщи пользователю. Свой orch-worktree можно оставить для следующей сессии (тогда вход = `sync`, без `--adopt`).
6. **Финализация (последний оркестратор, по команде пользователя).** Если что-то вливалось в `integration` (авто-fallback при грязном main) — сначала слить integration в main в чистом дереве main: `git merge --ff-only integration`, затем удалить ветку `integration`. Далее `multi off` (откажет, пока есть чужие in_progress) → закоммитить каталог трекера в main обычным способом → удалить orch-worktrees.

## Правила поверх /task

- **Никакого `lock`/`unlock`** — предохранитель multi-режима это owner-claim + mutex.
- **Статусы чужих задач не трогать.** `close`/`block`/`unblock` требуют `--owner`, CLI сверяет владельца и worktree; обход `--force` — только с подтверждения пользователя.
- **Пропущенный `sync --adopt` ловится:** worktree, не принявший multi-состояние, получает отказ на первой же операции с трекером (CLI видит `_MULTI` в ветке task-sync).
- **`new` атомарен:** задача считается созданной в момент коммита в task-sync; обрыв процесса не порождает дубль id. Дубли, занесённые руками, sync подсвечивает («⚠ ДУБЛЬ id»).
- **Не редактируй чужие спеки.** Правка спеки задачи = только её owner (или задача свободна и правка согласована).
- **Конфликт sync** («обе стороны меняли») — сигнал нарушения дисциплины владения. Останови работу, покажи пользователю обе версии (`git show task-sync:<файл>` vs локальная), только потом `sync --prefer`.
- **`merge-main` — единственный путь в main.** Ручной `git merge` в main из-под оркестратора запрещён: он вне глобального mutex и гонится с другими оркестраторами.
- **.gitignore:** `<tracker>/_LOCK`, `<tracker>/_MULTI`, `<tracker>/.sync-base` — runtime-состояние, в код-ветки не коммитится. `_SCHEMA` НЕ игнорируется — версия схемы синхронизируется и версионируется.
- **Schema-gate.** `_SCHEMA` синхронизируется между worktree. Отказ «трекер схемы vN, CLI понимает только vM» = твой `_cli.py` устарел относительно другого оркестратора: обнови `<tracker>/_cli.py` из каноника `.claude/skills/task/scripts/task.py` и повтори. Не понижай версию в `_SCHEMA` руками.
- **Отчёт об ошибке — простыми словами (как в `/task`),** включая multi-специфику: конфликт sync, отказ merge-main, schema-gate, чужой owner-claim. Пользователю: что сломалось, где, чем грозит, что дальше; без сырых логов — максимум одна решающая строка цитатой.
- **Автономия** — уровни те же, что в `/task` (раздел «Уровни автономии»). Уточнение для multi: в `autonomous` запрещены `multi off --force` и `unlock --force` (как и любой другой `--force`) — вместо force блокируй задачу и отчитайся пользователю.

## Ограничения (честно)

- Кросс-оркестраторные `depends`: `ready` видит done из task-sync — работает, но код зависимой задачи попадает к тебе только после её merge-main + подтяжки main в твой worktree (`git merge main` в orch-ветку).
- `verify --base main` при уехавшем main может показать чужие файлы в диффе — ветвись от свежего main или укажи `--base` явно (merge-base решает большинство случаев автоматически).
- **Уехавший main.** merge-main печатает предупреждение «main ушёл вперёд на N коммит(ов)» всегда, когда base опередил точку ветвления (чужие merge между твоим ветвлением и слиянием). Для задач с `risk>=2` в такой ветке перетест на свежем main — обязателен (процессно): подтяни main в `task/NNNN`, прогони тесты, только потом merge-main.
