# DESIGN.md — дизайн-лог скилла /task

> Дистиллят диалога проектирования (июль 2026). Назначение: контекст «почему так»
> для будущих правок скилла. Читать при изменении скилла, не при обычной работе.

## Происхождение
Порт скилла fable-ruki-agenty (github.com/serejaris/personal-corp-skills) с GitHub
Issues на локальный файловый трекер. Требования: без сети/gh, кросс-поддержка
Claude Code (терминал/VS Code/Desktop), Codex, Cursor.

## Ключевые решения и их причины

**Файл-на-задачу + мини-CLI (не единый BACKLOG.md, не SQLite, не готовые трекеры).**
Один файл = нет конфликтов при параллельной работе; markdown = человекочитаемый
дифф и возможность править спеку текстом; CLI без зависимостей (stdlib) = работает
везде. После `init` CLI копирует себя в `tasks/_cli.py` — проект самодостаточен,
все инструменты зовут один и тот же скрипт.

**_INDEX.md — генерируемый артефакт, не документ.** Перегенерация при каждой
мутации, руками не редактировать. Единственный писатель tasks/ — оркестратор
(исполнителям запрещено), поэтому индекс бесконфликтен по построению.
Замер на реальном проекте: ориентация по индексу в ~54 раза дешевле чтения спек.

**Статусы только через CLI**, иначе индекс разъезжается. open → in_progress →
done, плюс blocked.

**ready кодирует планирование детерминированно:** deps выполнены + files не
пересекаются с in_progress; предупреждает о пересечениях среди готовых.
Слабое звено — полнота files; заполнять через impact analysis (Codebase-Memory).

**Модели: дефолты в тексте скилла (haiku чтение / sonnet код), переопределение
аргументом вызова.** Конфиг-файл отвергнут сознательно (лишняя сущность).
В Codex/Cursor модель субагенту не навязать — только рекомендация.

**Режим исполнения: директива пользователя («параллельно»/«последовательно») —
высший приоритет, дефолт последовательный.** Параллельно → worktree обязателен
даже для двух агентов (ветки изолируют историю, НЕ рабочий каталог; общий
каталог = смешанные правки, гонка за git-индекс). Последовательно → основное
дерево, ветка task/NNNN. Ветка на задачу обязательна всегда: на ней держатся
verify и откат. Директива не отменяет безопасность: пересечения по files идут
последовательно в любом случае.

**verify — детерминированный контроль скоупа** (git diff main...task/NNNN против
files): ловит правки вне спеки и прикосновения к tasks/. Причина появления:
запреты конверта — инструкция, не механизм. Префикс tests/ разрешён по умолчанию
(--allow настраивается). Защита от коллизий = 4 слоя: ready (предотвращение),
worktree (изоляция), verify (контроль), merge-конфликт (обнаружение).

**Компакция архива** (`archive`). `_ARCHIVE.md` — отдельный от `_INDEX.md`
файл, а не просто «показывать архивные задачи в самом индексе». Цель индекса —
стоимость ориентации, а не полнота: если архивные записи (даже в свёрнутом
виде) продолжают накапливаться внутри `_INDEX.md`, файл всё равно растёт
линейно с историей проекта — то же исходное больное место. Разделение держит
`_INDEX.md` строго на размере живого набора задач; `_ARCHIVE.md` растёт, но
его читают только при явном интересе к истории, не при каждой ориентации.
Триггер ручной, а не автоматический внутри `close`: мутации трекера — только
явные и предсказуемые (тот же принцип, что и «Все мутации `tasks/` делает
только оркестратор»); автоматическая архивация в `close` спрятала бы решение
«эта задача теперь read-only» внутри рутинной операции закрытия. Выжимка
по умолчанию собирается оркестратором через `--summary` (у него есть контекст
приёмки целиком); CLI-fallback (первое предложение секции `## Приёмка`)
существует для пакетного `archive --done` и как страховка, но заведомо беднее
осмысленной выжимки. Prior art: механизм «memory decay» экосистемы beads
(автоматическое затухание давности задач в БД) — рассмотрен и отклонён: их
подход завязан на собственную БД поверх файлов, а не на markdown+git, который
и есть весь смысл этого трекера.

## Правила, рождённые из инцидентов

1. **Коммитить tasks/ ДО создания worktree** — worktree ответвляется от HEAD,
   незакоммиченная спека исполнителю не видна. (Найдено боевым тестом.)
2. **git worktree remove --force** — обычный remove падает на артефактах
   тестов (__pycache__ и т.п.); .gitignore обязателен. (Боевой тест.)
3. **Анти-фабрикация в приёмке:** каждый критерий — ссылка файл:тест,
   существование проверено grep-ом, прохождение — запуском. Причина: исполнитель
   дважды (smoke-тест и реальный прогон) отчитывался о несуществующих/незапущенных
   тестах. Заявленный, но не найденный тест = возврат задачи.
4. **Вся работа исполнителя закоммичена до отчёта** — незакоммиченное не
   существует для verify/merge. Оркестратор коммитит tasks/ после каждого close
   («task NNNN: приёмка») — git-история tasks/ = журнал конвейера.
5. **Serena активируется на пути СВОЕГО worktree**, не основного дерева.
6. **Serena — пропорционально, не ритуально:** тулы стоят контекста при каждом
   спавне; мелкие правки — обычным edit. Оркестратор может предписать в спеке.
7. **Правило проводки** — дважды за один день (инциденты 0011, 0012 боевого
   прогона) исполнитель сдал модуль с новым API без подключения в app_main:
   «deferred wire, trivial follow-up». Фича мертва в прошивке, unit-тесты
   зелёные. Корень — дыра декомпозиции: `files` не включал точку интеграции,
   verify исполнителя туда и не пустил бы. Отсюда: потребитель — в `files`,
   e2e-наблюдаемость — в критерии.
8. **Правило общих ресурсов** — исполнитель параллельной задачи занял
   namespace, уже закреплённый другой задачей волны с противоположной
   семантикой (factory-reset); `ready` пропустил — файлы не пересекались.
   Отсюда: аллокации приколачивать конкретными значениями в спеке,
   при параллельной волне — сводная карта аллокаций.
9. **Правило симметричных кейсов** — спека флаговала гонку «миграция vs
   reset» для reg-1001 (находка ревью предыдущей задачи), исполнитель закрыл
   ровно её; зеркальный reg-1000 с той же гонкой пропустил — REJECT
   (ревьюер доказал экспериментально: legacy.addr=200 → addr=7, контракт
   «reset сохраняет адрес» сломан). Исполнитель чинит букву спеки —
   перечислять симметричные кейсы обязан оркестратор.
10. **Правило зачистки при удалении** — после удаления компонента логика
   принята с первого захода, но по репо остались present-tense описания
   компонента как живого (README, комментарии заголовков) — REJECT по
   критерию «grep по репо»; `files` расширялся постфактум. Отсюда: доки
   и комментарии — в `files` и критерии сразу при декомпозиции.

Причина Serena-решения в спеке и отчёте (правка после прогона KVDB v3):
за 12 задач Serena не была предписана ни разу и ни разу не всплыла в
отчётах — правило пропорциональности работало вслепую, без данных.
Фиксация причины («не нужна: файлы маленькие» / «нужна: ренейм») даёт
обратную связь: где отказ стоил итерации, где подключение не окупилось.

## Экономика токенов

**spent: sonnet(2):34k, serena(2):2k** — скобки = число спавнов (для MCP-ключей —
спавнов с подключённым инструментом, НЕ вызовов: константа = цена загрузки тулов
в контекст). Только цифры из счётчиков инструмента; нет счётчика → поле
пропускается, не выдумывается. Учитываем потраченное, не «сэкономленное»
(экономия = контрфактуал, измерима только сравнением с baseline через /cost).

**Калибровка:** тривиальный субагент на конфигурацию → input = константа спавна;
serena = разность (с − без). Хранится в _CONSTANTS.md РЯДОМ С ЗАДАЧАМИ, не в
папке скилла (CLI после init не знает путь скилла; у Codex/Cursor папки нет;
состав инструментов — свойство проекта). stats раскладывает: работа
(модельно-инвариантна) vs инициализация (спавны × константа). Доля инициализации
стабильно >30% → задачи слишком мелкие. Оценка не учитывает кэш и разное число
итераций моделей.

## Технические границы

- Каталог задач переименовываем (tasks/ → agent_tasks/): CLI привязан к своему
  расположению; verify берёт префикс динамически. Переименование — только в
  паузе (нет in_progress/worktree), через git mv.
- Нумерация не имеет потолка: регэксп \d{4,}, сортировка числовая
  (баг «после 9999» найден и закрыт: задачи становились невидимками).
- grep -i не фолдит кириллицу в C-локали — приёмочные проверки русского текста
  без расчёта на регистронезависимость.
- Serena в параллельном режиме — один пользователь за раз: MCP-серверы общие
  на сессию, activate_project — глобальное состояние сервера; worktree-контексты
  он не изолирует. Отсюда правило «Serena максимум у одного параллельного
  исполнителя» в разделе MCP. Многоагентная параллель с Serena боевым тестом
  не гонялась (инцидент №5 — про один worktree).

## Кросс-поставка
SKILL.md — открытый стандарт: Claude Code читает из .claude/skills/task/,
Codex — из .agents/skills/task/ (нативно, вызов $task). sync_rules.py — запасной
путь: ядро (между core:start/end) → AGENTS.md (always-on контекст) и
.cursor/rules/task.mdc. Симлинк .agents/skills/task → ../../.claude/skills/task
даёт один источник правды.

## Заимствования (2026-07-03)

Из разбора multi-model-pipeline-skill (github.com/alexxety/multi-model-pipeline-skill),
их сила — независимость ревью, наш слабый участок на тот момент. Взято:
- **Долг из ревью** — у нас не-блок находки умирали в приёмочных записях
  (0009 нет теста CORRUPT-ветки, 0011 e2e только sync, 0012 нет теста
  eviction-vs-in-flight, 0013 запаркованные доки) при живом трекере рядом.
- **Vacuous-pass контроль** — анти-фабрикация ловила «теста нет/не гонялся»,
  но не «тест зелёный и проверяет пустоту»; новый guard-тест обязан уметь падать.
- **Конверт ревьюера** (refute-framing: претензия → опровергни, категории
  охоты, CONFIRMED file:line) — острее, чем «проверь критерии».
- **Ревью merge-коммитов оркестратора** — слитый результат не видел ни один
  исполнитель.
- **Правило гипотез** — непроверенное в спеке помечать, не подавать фактом
  (риск класса 0007: спека подала кейс reg-1001 фактом, соседний пропущен).
- **FP-список ревьюера** — из Anthropic code-review plugin (канонический
  список ложных срабатываний); numeric confidence-scorer 0–100 отклонён
  (echo-риск finder+scorer, бинарный refutation-вердикт лучше для конвейера).

Рассмотрено и отложено: кросс-вендорное ревью (Codex read-only против
Claude-оркестратора) — требует внешний CLI и денег; same-vendor-ревью ловило
все известные итерации, кросс-вендор страхует невидимые false-accept.
Кандидат: опциональная эскалация для high-risk задач (BL/flash/concurrency).
