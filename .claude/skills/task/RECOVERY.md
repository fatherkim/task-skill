# RECOVERY — рецепты восстановления трекера задач

Справочник «симптом → диагностика → точные команды → когда звать человека» для
нештатных ситуаций конвейера. Сообщения об ошибках `task.py` ссылаются сюда
хвостом `Рецепт: RECOVERY.md#<якорь>`; открой соответствующий раздел.

Общие принципы:

- Ничего не удаляй вслепую. Сначала диагностика (`doctor`, `multi status`, `git`),
  потом действие.
- Каталог трекера в примерах — `tasks/` (в этом репозитории — `agent_tasks/`; в
  командах подставляй фактическое имя, оно же в путях `_cli.py`).
- `doctor` (read-only) — первая команда при любом подозрении: `python3 tasks/_cli.py doctor`.

## <a id="lock"></a>Чужой `_LOCK` (трекер залочен другой сессией)

**Симптом.** Мутирующая команда падает: `ОТКАЗ: трекер залочен <owner> (...)`.
`doctor` в проверке 5 показывает `_LOCK: owner=<чужой>`.

**Диагностика.**
```
python3 tasks/_cli.py lock          # покажет owner и возраст замка
python3 tasks/_cli.py doctor        # проверка 5: возраст, «возможно протух» при >24ч
```
- Если owner — это твоя же сессия (совпадает твоя метка) — просто передай
  `--owner <метка>` или выставь `TASK_OWNER`.
- Если owner чужой и замок свежий (минуты/часы) — **работает другой оркестратор.
  СТОП.** Не запускай конвейер параллельно в single-режиме.

**Команды.**
```
# твоя же метка — укажи владельца, гейт пропустит:
python3 tasks/_cli.py new "..." --owner <твоя-метка>

# замок протух (>24ч, автор точно не активен) — сними и захвати:
python3 tasks/_cli.py unlock --owner <твоя-метка>        # свой
python3 tasks/_cli.py unlock --force                     # чужой протухший
python3 tasks/_cli.py lock --owner <твоя-метка>
```

**Когда звать человека.** Замок свежий и ты не знаешь, кто его держит; либо два
оркестратора реально нужны одновременно — тогда не `--force`, а переход в
multi-режим (`multi on`) с worktree на каждого. Осознанный `--force`/перехват
чужого свежего замка — только по явному подтверждению пользователя.

## <a id="sync-conflict"></a>`sync: конфликт — обе стороны меняли`

**Симптом.** `sync` (или любая multi-мутация) падает:
`sync: конфликт — обе стороны меняли: <файл(ы)>`. При дисциплине `start --owner`
(один файл задачи мутирует ровно один оркестратор) такого быть не должно —
значит два worktree тронули один и тот же файл трекера.

**Диагностика.** Сравни свою версию файла с версией ветки `task-sync`:
```
git show task-sync:0001-slug.md          # версия в общей ветке
cat tasks/0001-slug.md                    # твоя локальная версия
git log --oneline task-sync -- 0001-slug.md
```
Реши, какая сторона права по СОДЕРЖАНИЮ (кто владелец задачи по `owner:`; чьё
изменение легитимно). Конфликт всегда по конкретному файлу задачи или служебному
файлу — не по всему трекеру.

**Команды.** После того как решил, чью версию принять:
```
python3 tasks/_cli.py sync --prefer theirs   # взять версию task-sync
python3 tasks/_cli.py sync --prefer ours      # оставить локальную, затолкать в ветку
```
`--prefer` применять ТОЛЬКО после ручного разбора обеих версий — он глушит
трёхстороннее слияние и берёт одну сторону целиком.

**Когда звать человека.** Если обе версии несут нужные, но разные правки одного
файла (реальная потеря данных при выборе любой стороны) — слей их руками в
`tasks/<файл>`, затем `sync --prefer ours`. Систематические конфликты по одному
файлу = нарушена дисциплина владения: эскалируй, разберись, кто мутирует чужую
задачу.

## <a id="dup-id"></a>Дубли id (две задачи с одним номером)

**Симптом.** `doctor` проверка 2: `дубли id: 0001 (0001-alpha.md, 0001-beta.md)`;
либо `sync` печатает блок `⚠ ДУБЛЬ id ...`. Обычно — гонка `new` в двух worktree
до синхронизации, либо ручное копирование файла задачи.

**Диагностика.** Кто настоящий владелец номера — по истории ветки `task-sync`:
```
python3 tasks/_cli.py doctor                  # проверка 2 перечислит файлы-дубли
git log --oneline task-sync -- 0001-*.md      # кто закоммитил номер первым
```
Первый в истории `task-sync` — настоящий; второй нужно переномеровать.

**Команды.** Переномеровать второй файл на следующий свободный id:
```
git mv tasks/0001-beta.md tasks/0007-beta.md              # 0007 — следующий свободный
# правь `id:` в frontmatter переименованного файла на 0007
grep -rn '0001' tasks/*.md                                # поправь depends, ссылавшиеся на дубль
python3 tasks/_cli.py index
python3 tasks/_cli.py sync                                 # multi: разослать исправление
```

**Когда звать человека.** Если на дубль уже ссылаются `depends` других задач и
неясно, какую из двух они имели в виду; либо оба файла содержат реальную,
несовпадающую работу (обе задачи настоящие) — тогда решение о номерах и связях
принимает оркестратор/пользователь, а не автоматика.

## <a id="dirty-main"></a>Грязный `main` при `merge-main`

**Симптом.** `merge-main ... ГРЯЗНОЕ` — базовая ветка (`main`) выложена в
worktree с незакоммиченными изменениями, поэтому безопасный merge туда невозможен.

**Диагностика.**
```
git worktree list                             # где checked out main
git -C <path-к-main-worktree> status          # что там незакоммичено
```

**Команды (в порядке предпочтения).**
```
# 1) прибраться в main-worktree: закоммить или спрятать
git -C <main-worktree> stash        # или commit
python3 tasks/_cli.py merge-main <N>

# 2) дать авто-fallback: merge уходит в общую ветку integration (без --no-fallback)
python3 tasks/_cli.py merge-main <N>          # по умолчанию fallback включён
#    финализация позже: git merge --ff-only integration  (в чистом дереве main)

# 3) явно мержить в integration
python3 tasks/_cli.py merge-main <N> --base integration
```

**Когда звать человека.** Незакоммиченное в main — чужая незавершённая работа,
которую нельзя ни спрятать, ни выбросить без владельца. Не делай `stash`/`reset`
за другого.

## <a id="integration"></a>Stale `integration` (ветка отстала / финализация)

**Симптом.** `doctor` проверка 9: `integration: main не влит (stale/diverged)`.
Ветка `integration` копила fallback-мержи, но `main` ушёл вперёд и больше не
является её предком — прямой `git merge --ff-only integration` не пройдёт.

**Диагностика.**
```
python3 tasks/_cli.py doctor                             # проверка 9
git merge-base --is-ancestor main integration; echo $?   # 0 = integration содержит main
git log --oneline main ^integration                       # чего в main нет в integration
```

**Команды.**
```
# подтянуть main в integration (любой merge-main выравнивает integration автоматически):
python3 tasks/_cli.py merge-main <любая-готовая-задача>
# затем финализировать в main:
git switch main
git merge --ff-only integration
git branch -d integration            # ветку можно удалить после финализации
```
Если `integration` и `main` разошлись содержательно (не только отставание) —
merge-main сам попытается слить их merge-tree; при конфликте он подскажет
`merge-main --branch <...> --base integration --resolve`.

**Когда звать человека.** Разошлись содержательно и авто-слияние падает на
конфликте, который нельзя разрешить механически (обе ветки правили один код).

## <a id="worktree-remove"></a>`git worktree remove` падает на артефактах

**Симптом.** После merge/резолва снятие временного worktree отказывает:
`fatal: '<path>' contains modified or untracked files, use --force to delete it`
(обычно это площадка `--resolve`: `_merge/NNNN`, либо worktree исполнителя со
сборочными артефактами).

**Диагностика.**
```
git worktree list
git -C <path> status --porcelain      # что именно мешает (артефакты сборки, .o, логи)
```
Убедись, что «модифицированное/untracked» — это МУСОР (артефакты), а не
несохранённая работа.

**Команды.**
```
# если внутри только артефакты — снять принудительно:
git worktree remove --force <path>
git branch -D _merge/NNNN             # ветку площадки резолва — тоже
git worktree prune                    # подчистить записи об исчезнувших worktree
```

**Когда звать человека.** Если `status` показывает НЕ артефакты, а осмысленные
изменения кода — это несмерженная работа; сначала разберись, куда её деть
(commit/merge), и только потом удаляй.

## <a id="schema"></a>`_SCHEMA` новее CLI (`tasks/_cli.py` устарел)

**Симптом.** Любая команда падает: `ОТКАЗ: трекер схемы vN, CLI понимает только
vM — обнови tasks/_cli.py из каноника`. Кто-то обновил канонический `task.py`
(новая версия схемы), а копия `tasks/_cli.py` в репозитории осталась старой.
`doctor` проверка 1 показывает `_SCHEMA vN новее CLI`.

**Диагностика.**
```
python3 tasks/_cli.py doctor          # проверка 1: версия трекера vs CLI
cat tasks/_SCHEMA                      # версия схемы трекера
```

**Команды.** Обновить копию CLI из канона (`.claude/skills/task/scripts/task.py`):
```
cp .claude/skills/task/scripts/task.py tasks/_cli.py
python3 tasks/_cli.py doctor          # проверка 1 должна стать ✓
```
`_SCHEMA` руками НЕ понижать — гейт защищает формат; чинить нужно CLI, а не число.

**Когда звать человека.** Если канонического `task.py` подходящей версии нет под
рукой (обновление схемы принёс другой оркестратор и его ещё не влили) — дождись
влития канона или запроси его; не редактируй `_SCHEMA`, чтобы «обойти» гейт.

## <a id="serena-worktree"></a>Serena активирована не в том worktree

**Симптом.** В multi-режиме каждый оркестратор/исполнитель — в своём git worktree.
Если MCP-инструмент индексации кода (Serena и подобные) активирован на КОРНЕВОЙ
репозиторий или на чужой worktree, правки/ссылки на символы уходят не в тот
рабочий каталог: «символ не найден», правки в неожиданном дереве, дифф пуст там,
где ждёшь изменения.

**Диагностика.**
```
git rev-parse --show-toplevel         # фактический корень ТЕКУЩЕГО worktree
git worktree list                      # все worktree репозитория
```
Сверь корень активной Serena-сессии с тем worktree, где реально идёт работа над
задачей (`worktree:` в frontmatter задачи; `doctor` проверка 7 подсветит
несуществующий/чужой worktree у in_progress-задач).

**Команды / действия.**
- Переактивируй инструмент на корень нужного worktree (путь из
  `git rev-parse --show-toplevel` внутри рабочего каталога исполнителя).
- Не полагайся на символьные правки, пока корень не совпал; для точечной правки
  файла используй прямое редактирование по абсолютному пути worktree.
- После переактивации проверь: правка попадает в дифф ветки задачи
  (`python3 tasks/_cli.py verify <N>`).

**Когда звать человека.** Если инструмент упорно цепляется к чужому worktree и
переактивация не помогает — останови работу и эскалируй: продолжать с
рассинхронизированным индексом = писать код мимо ветки задачи.

## Агент умер / отчёт потерян {#agent-death}

Симптом: исполнитель/ревьюер завершился без отчёта, прислал мусор, или
сессия с агентами погибла.

1. Агент жив (глюк, пустой ответ) → SendMessage ему: «пришли дайджест и путь
   к отчёту» — чаще всего срабатывает (прецедент: ревью 0066).
2. Агент мёртв → читай .reports/{ID}-*.md напрямую.
3. Файла нет → следы в git: `git log --oneline -5` и `git status` на ветке
   task/NNNN. Частичная работа часто корректна.
4. Переспавн — конверт-преамбула:
   «Твой предшественник начал задачу {ID} и мог погибнуть на середине.
   Прежде чем делать что-либо: git log -5 и git status на ветке task/{ID},
   прочитай .reports/{ID}-coder.md, если есть. Частичную корректную работу
   прими и продолжи — не переделывай с нуля. Спека: {путь}.»
   Дальше — обычный конверт исполнителя.

Не переделывать с нуля без аудита следов: выброшенная наполовину сделанная
работа — типовая потеря токенов.
