# task — руки-агенты с локальным трекером

Мультиагентный конвейер разработки. Главная сессия — оркестратор — **сама код не пишет**: она декомпозирует работу на задачи, хранит спеки в `tasks/`, диспатчит исполнителей указателем на файл задачи и принимает результат. Трекер — локальные markdown-файлы + мини-CLI (никакого GitHub Issues и сети), поэтому ядро конвейера работает в Claude Code / Claude Desktop / VS Code, а через генератор правил — также в Codex и Cursor.

## Модели субагентов

Дефолты: **читающие/разведывательные субагенты — haiku**, **кодящие исполнители — sonnet**.

Приоритет переопределения:
1. Аргументы вызова скилла — высший приоритет. Примеры: `/task кодинг на opus`, `coder=opus reader=sonnet`, «исполнителей запускай на opus». Распарсь свободный текст и применяй указанные модели ко всем последующим запускам в этой сессии.
2. Если ничего не указано — дефолты выше.

В Codex/Cursor модель субагенту принудительно задать нельзя — там это рекомендация пользователю при открытии новой сессии.

## Первый запуск в репозитории

```bash
python3 ${CLAUDE_SKILL_DIR}/scripts/task.py init        # создаст tasks/, скопирует CLI в tasks/_cli.py
python3 ${CLAUDE_SKILL_DIR}/scripts/sync_rules.py       # (опционально) сгенерирует AGENTS.md и .cursor/rules для Codex/Cursor
```

(`${CLAUDE_SKILL_DIR}` резолвится в Claude Code автоматически; вне Claude Code подставь путь к папке скилла вручную.)

После `init` проект самодостаточен: все инструменты дальше зовут `python3 tasks/_cli.py <команда>`. Если `tasks/_cli.py` уже есть в репозитории — `init` не нужен.

## Хранилище задач

- Одна задача = один файл `tasks/NNNN-slug.md`. Frontmatter: `id, status, title, files, depends, created`.
- Имя каталога по умолчанию `tasks/`, но его можно переименовать (например, `agent_tasks/`): CLI привязан к каталогу, где лежит `_cli.py`, а не к имени. Ориентир для агентов — каталог с `_cli.py` и `_INDEX.md`; его фактическое имя использовать во всех командах и конвертах вместо `tasks/`. Переименование делать только в паузе: нет задач `in_progress` и открытых worktree, через `git mv` + коммит.
- **Тело файла — спека и единственный источник требований.** Не пересказывай спеку в промптах — передавай путь к файлу. Историю изменений хранит git.
- Статусы: `open → in_progress → done`, плюс `blocked`. Менять статусы **только через CLI** — иначе разъедется индекс.
- `files` во frontmatter — честный список затрагиваемых путей: от него зависит планирование параллельности. Уточняй его при изменении скоупа (правкой frontmatter + `python3 tasks/_cli.py index`).
- `tasks/_INDEX.md` — генерируемый обзор (таблица id/статус/файлы/зависимости). Читай его для ориентации вместо сканирования всех файлов. **Руками не редактировать**; после любых сомнений — перегенерировать.
- Файлы с префиксом `_` задачами не являются.
- Все мутации `tasks/` делает только оркестратор. Исполнители каталог `tasks/` не трогают вообще.

## Мини-CLI

Все команды: `python3 tasks/_cli.py <cmd>` из корня репозитория.

| Команда | Что делает |
|---|---|
| `new "Заголовок" --files a.py,b.py --depends 0003` | создать задачу с шаблоном спеки |
| `list [--status open]` | таблица задач |
| `view N` | показать спеку задачи |
| `ready` | что можно диспатчить: deps выполнены, файлы не заняты in_progress; предупреждает о пересечениях между готовыми |
| `start N` | взять в работу |
| `block N --reason "..."` / `unblock N` | заблокировать / снять блокировку |
| `verify N [--base main] [--allow tests/]` | сверить дифф ветки `task/NNNN` с заявленными `files`: ловит правки вне скоупа и любые прикосновения к `tasks/`; exit 1 при нарушении |
|  `close N --note "..." [--spent "sonnet(2):34k,serena(2):2k"]` | закрыть с записью приёмки; `--spent` — фактический расход токенов по моделям |
|  `stats [--init "..."]` | сводка: задачи по статусам, расход по моделям, во сколько раз индекс дешевле чтения всех спек |
| `calibrate [--set "sonnet:12k,serena:8k"]` | показать/сохранить цены спавна в `_CONSTANTS.md`; stats подхватывает их автоматически |
| `index` | перегенерировать `_INDEX.md` |

Каждая мутирующая команда сама перегенерирует индекс.

## Конвейер

1. **Декомпозиция.** Разбей работу на компактные, по возможности независимые задачи. Для каждой: `new`, затем заполни спеку в теле файла (Задача / Контекст / Критерии приёмки / Вне скоупа). Критерии — проверяемые, без «сделать хорошо». Если подключён Codebase-Memory — прогони impact analysis, чтобы список `files` был полным, а не угаданным.

   **Правило проводки.** Задача, добавляющая новый API/фичу, обязана включать точку интеграции (потребителя: main loop, init-путь, вызывающий модуль) в `files` и критерий приёмки «фича наблюдаема через реальный boot/e2e-путь» (живой вызов, ненулевой счётчик), а не только unit-тесты. Модуль без проводки = мёртвый код в прошивке; «подключим потом» без отдельной задачи в трекере — не принимается.

   **Правило общих ресурсов.** Аллокации из общих пространств (namespace, ключевые диапазоны, адреса регистров, ID) приколачивай в спеке конкретными значениями при декомпозиции — не оставляй выбор исполнителю: `ready` ловит пересечения по файлам, но не по логическим ресурсам. При параллельной волне — сводная карта аллокаций волны в спеках всех затронутых задач.
2. **Планирование.** `ready` показывает, что можно запускать. Задачи с пересекающимися `files` — строго последовательно; непересекающиеся — можно параллельно (в отдельных worktree). Спорные развилки фиксируй в спеке вопросом к пользователю до диспатча.

   **Режим исполнения.** Приоритет: (1) явное указание пользователя — в вызове скилла или по ходу сессии («работай параллельно», «дальше последовательно») — действует до отмены; (2) иначе решает оркестратор: по умолчанию последовательно, параллель — когда задачи долгие и `ready` подтверждает совместимость. Механика следует из режима автоматически: параллель → worktree на каждую задачу; последовательно → основное дерево, ветка `task/NNNN`. Директива «параллельно» — это предпочтение, а не отмена безопасности: пересекающиеся по `files` и связанные `depends` задачи идут последовательно в любом случае — параллелится только то, что `ready` показывает совместимым, остальное честно сообщается пользователю.
3. **Диспатч указателем.** Перед запуском исполнителя: `start N`, затем **закоммить `tasks/` в текущую ветку** — worktree ответвляется от HEAD, и незакоммиченные спеки исполнитель в своём дереве не увидит. В промпт исполнителю — конверт (ниже) с путём к файлу задачи, а не текст спеки.
4. **Исполнение.** Исполнитель работает строго в рамках конверта. При блокере — останавливается и отчитывается; оркестратор делает `block N --reason`.
5. **Приёмка.** Сначала `verify N` — детерминированный контроль скоупа (дифф против `files`, прикосновения к `tasks/`); при провале — либо возврат исполнителю, либо осознанное обновление `files` в спеке (+ `index`). Затем оркестратор проверяет дифф против критериев приёмки (это можно поручить читающему субагенту). **Ссылкам исполнителя не верить на слово:** каждый закрытый критерий подтверждается конкретной ссылкой файл:тест, чьё существование проверено (grep по имени в диффе/дереве), а прохождение — фактическим запуском; заявленный, но не найденный тест — фабрикация и автоматический возврат задачи. Успех: `close N --note "..."`, слить ветку, затем закоммитить `tasks/` (файл задачи со статусом и приёмкой + индекс) с сообщением `task NNNN: приёмка` — каждый цикл задачи завершается коммитом, история `tasks/` в git и есть журнал конвейера. Провал: конкретные замечания исполнителю, статус остаётся `in_progress`.

   **Учёт токенов (опционально).** При закрытии: `--spent "sonnet(2):34k, haiku(1):6.2k, serena(2):2k"` — в скобках число спавнов модели (для MCP-ключей вроде serena — число спавнов с подключённым инструментом; без скобок считается 1). Цифры — только из счётчиков инструмента (результат Task-вызова, `/cost`); если счётчика нет, поле пропускается, не выдумывается. `stats` агрегирует расход и при наличии калибровки автоматически раскладывает его на работу и инициализацию. Это учёт потраченного, а не «сэкономленного».

   **Калибровка (`/task calibrate`).** По запросу «откалибруй цены спавна» оркестратор: (1) спавнит тривиального субагента («ответь одним словом OK») на каждую используемую конфигурацию — читатель haiku без MCP, кодер sonnet без MCP, кодер sonnet с Serena, при необходимости opus и codebase_memory; (2) берёт input-токены каждого из счётчиков Task-результатов; (3) ключи моделей = базовый спавн, ключи инструментов = разность (с инструментом − без); (4) сохраняет: `python3 tasks/_cli.py calibrate --set "sonnet:12k,haiku:4k,serena:8k"`. Константы лежат в `_CONSTANTS.md` рядом с задачами (не в папке скилла: CLI после init не знает её путь, у Codex/Cursor её нет, а состав инструментов — свойство проекта) и версионируются git. Разложение: работа = расход − спавны × константа (модельно-инвариантная часть — та же спека и файлы стоили бы примерно столько же другой модели); оценка не учитывает кэширование префикса и разное число итераций у моделей. Доля инициализации стабильно выше ~30% — задачи слишком мелкие, укрупняй декомпозицию. Перекалибровка — при смене версии инструментов, состава MCP или системного промпта; `stats --init "..."` разово переопределяет константы без записи.

Защита от коллизий двух агентов на одном файле — три слоя: `ready` не даёт диспатчить пересекающиеся по `files` задачи (предотвращение); параллельные исполнители сидят в разных worktree, физически один файл одновременно не редактируется, коллизия откладывается до мерджа (изоляция); `verify` + merge-конфликт на приёмке ловят выход за скоуп и недекларированные пересечения (обнаружение). Слабое звено — полнота `files`, поэтому заполняй его через impact analysis, а не наугад.

## MCP-инструменты (если подключены)

Если в сессии доступны Serena и/или Codebase-Memory — используй их по разделению ролей ниже. Если не подключены — работай обычными средствами, конвейер от них не зависит.

Use Serena for:
- symbol-level edits
- refactoring
- rename/move operations
- reading exact symbol bodies
- finding definitions/references through LSP
- applying changes safely

Use Codebase-Memory for:
- call graph questions
- impact analysis
- architectural overview
- dependency chains
- "where does this flow start/end?"
- cross-module relationships
- persistent codebase memory across sessions

**Do not query both tools for the same lookup** unless the first result is incomplete or contradictory.

**Пропорциональность, не ритуал.** Инструменты Serena стоят контекста при каждом спавне субагента (загрузка определений + активация проекта), поэтому: точечные правки в маленьких файлах — обычным edit; Serena подключай, когда symbol-level реально выигрывает — большие файлы (чтение тела символа вместо всего файла), ренеймы/перемещения с множеством references, изменение сигнатур, незнакомый код. Оркестратор может указать это прямо в спеке («правки через Serena» / «Serena не нужна») — тогда исполнитель не решает сам.

Привязка к этапам конвейера:
- **Декомпозиция (оркестратор / читающий субагент):** Codebase-Memory — impact analysis и dependency chains, чтобы честно заполнить `files` во frontmatter и выставить `depends`. Точность `files` = качество планирования параллельности.
- **Исполнение (кодящий субагент):** Serena — правки на уровне символов вместо сырых патчей; definitions/references перед изменением сигнатур. При работе в worktree Serena активируется на пути этого worktree, не на основном дереве.
- **Параллельный режим — Serena максимум у одного исполнителя одновременно.** MCP-серверы общие на сессию: Serena-сервер один, `activate_project` — его глобальное состояние, два параллельных исполнителя перещёлкивают проект друг у друга и символьные операции уходят не в то дерево. Оркестратор при параллельном диспатче даёт Serena одной задаче, остальным пишет в спеку «Serena не нужна», либо ставит Serena-тяжёлые задачи (ренеймы, смена сигнатур) последовательно.
- **Приёмка (читающий субагент):** Serena find_references — проверить, что изменённые символы не сломали вызовы за пределами диффа; Codebase-Memory — если критерий приёмки затрагивает архитектуру.

## Конверт исполнителя (шаблон промпта)

```
Ты — исполнитель задачи {ID}.
1. Прочитай спеку: {путь к tasks/NNNN-*.md}. Спека — единственный источник требований.
2. Рабочее место: {worktree/ветка}. За его пределы не выходи.
3. Запрещено: править что-либо в tasks/ (включая свою спеку и _INDEX.md);
   мержить в main; трогать файлы вне секции «Контекст»; расширять скоуп;
   менять зависимости проекта, если этого нет в спеке явно.
4. Коммиты с префиксом "task {ID}: ...". Вся работа должна быть закоммичена
   до финального отчёта: незакоммиченные изменения не видны ни verify, ни merge —
   их не существует для приёмки. Рабочее дерево перед отчётом — чистое
   (кроме игнорируемых артефактов).
5. Блокер — остановись и опиши причину. Не обходи его самостоятельно.
6. Финальный отчёт: что сделано; какие критерии приёмки закрыты (по пунктам);
   что не удалось и почему.
7. Если подключены Serena / Codebase-Memory — следуй правилам раздела
   «MCP-инструменты»: правки через Serena, анализ связей через Codebase-Memory,
   не дублируй один и тот же lookup в оба инструмента.
   ВАЖНО: перед первой правкой активируй Serena на пути СВОЕГО worktree
   ({worktree}), а не основного дерева репозитория — иначе правки уйдут не туда.
```

## Режим без субагентов

Если субагентов нет (Codex, Cursor, одиночная сессия): роли оркестратора и исполнителя исполняются последовательно в одной сессии либо в отдельных сессиях/вкладках — по одной задаче за раз. Конверт и запреты действуют без изменений: взяв роль исполнителя, не трогай `tasks/`; вернувшись в роль оркестратора — проведи приёмку и только тогда меняй статусы.

## Claude Code / Desktop: механика диспатча

- Исполнители — субагенты через Task tool: читающие на haiku, кодящие на sonnet (или модели из переопределения при вызове).
- Правило выбора: **параллельные исполнители — обязательно в отдельных worktree, даже если их всего два** (ветки изолируют историю, но не рабочий каталог: два агента в одном дереве смешивают правки, гонятся за git-индекс и ломают verify). **Последовательные исполнители worktree не требуют** — каждый работает в основном дереве на своей ветке `task/NNNN`, по одному за раз. Ветка на задачу обязательна в любом режиме: без неё нет `verify` и отката провальной работы.
- Создание worktree для параллельной задачи:
  ```bash
  git worktree add ../<repo>-wt-NNNN -b task/NNNN
  ```
- После приёмки: слить `task/NNNN`, затем `git worktree remove --force ../<repo>-wt-NNNN` — обычный `remove` упадёт на артефактах тестов/сборки (`__pycache__` и т.п.), которые исполнитель оставил в дереве. Убедись, что `.gitignore` покрывает такие артефакты, иначе они попадут в коммиты исполнителей.
- Проверку критериев приёмки по диффу поручай читающему субагенту (haiku): вход — путь к спеке + дифф, выход — вердикт по каждому критерию.

## Кросс-форматная поставка

Канонический текст — этот SKILL.md. SKILL.md — открытый стандарт (agentskills.io), и Codex CLI читает его нативно: та же папка скилла кладётся в `<repo>/.agents/skills/task/`, вызов — `$task` или автотриггер по description. Конвертация для Codex не нужна.

`scripts/sync_rules.py` — запасной путь: раскладывает инструментонезависимое ядро (между маркерами `core:start/core:end`) в `AGENTS.md` (всегда загружаемый контекст Codex — в отличие от скилла, который подгружается по совпадению) и `.cursor/rules/task.mdc` (Cursor). После правок ядра перезапусти `sync_rules.py` в целевом репозитории — блок в AGENTS.md обновится по маркерам, остальное содержимое файла не пострадает.
