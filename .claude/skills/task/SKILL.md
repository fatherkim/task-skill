# task — руки-агенты с локальным трекером

Мультиагентный конвейер разработки. Главная сессия — оркестратор — **сама код не пишет**: она декомпозирует работу на задачи, хранит спеки в `tasks/`, диспатчит исполнителей указателем на файл задачи и принимает результат. Трекер — локальные markdown-файлы + мини-CLI (никакого GitHub Issues и сети), поэтому ядро конвейера работает в Claude Code / Claude Desktop / VS Code, а через генератор правил — также в Codex и Cursor.

Быстрый старт — `QUICKREF.md` (1 страница: цикл single, 6 шагов multi, топ-5 ошибок).

## Модели субагентов

Дефолты по ролям — «дешевейшая модель, надёжно закрывающая задачу»; эскалация
(лестница, п.5 Конвейера) = шаг вправо по строке. В Claude Code уровень effort
задаётся ТИПОМ агента (.claude/agents/<model>-<effort>.md: fable/opus/sonnet ×
low/med/high, opus-xhigh, haiku) — живому агенту effort не поменять, только
переспавном.

| Роль | Дефолт | Сложнее | Потолок |
|---|---|---|---|
| Читающий/разведка (факты, дифф-чек критериев) | haiku | sonnet-low | sonnet-med |
| Кодер: механика (доки, конфиги, шаблонные правки) | sonnet-med | sonnet-high | — |
| Кодер: логика (код, тесты, нетривиальные правки) | sonnet-high | opus-med / fable-med* | opus-xhigh |
| Ревьюер | opus-high | opus-xhigh (core-протокол, критичный код) | opus-xhigh |
| Верификатор-прогон (выполнить DoD-команду, отчитаться) | haiku | sonnet-low | — |

*fable-med — тонкая семантика (git-внутренности, конкурентность, выдержанный
тон чужого текста); opus — глубина на объёме. Ревьюер никогда не слабее кодера
той же задачи. В Codex/Cursor модель не задать — таблица становится
рекомендацией пользователю при открытии сессии.

Приоритет переопределения:
1. Аргументы вызова скилла — высший приоритет. Примеры: `/task кодинг на opus`, `coder=opus reader=sonnet`, «исполнителей запускай на opus». Распарсь свободный текст и применяй указанные модели ко всем последующим запускам в этой сессии.
2. Если ничего не указано — дефолты выше.

## Уровни автономии

Уровень задаётся в вызове скилла — `автономия: interactive|supervised|autonomous` (свободный текст тоже парсится: «работай автономно», «спрашивай на всё») — и действует до конца сессии.

- **interactive** (дефолт): `AskUserQuestion` на все `--force`/деструктивные операции и на каждое срабатывание circuit breaker (3-й возврат). Ничего необратимого без явного согласия пользователя.
- **supervised**: `--force`-операции — только после вопроса пользователю; остальное (обычные возвраты, block, close, merge-main без конфликта) — молча, с итоговым отчётом. Circuit breaker — уведомить в отчёте, но можно продолжать по решению оркестратора.
- **autonomous**: вопросов нет, отчёт в конце. Все `--force`-операции **ЗАПРЕЩЕНЫ** — вместо force-обхода задача блокируется (`block N --reason "нужен force: ..."`) и выносится в отчёт. Оркестратор не перехватывает чужой lock, не снимает owner-guard, не давит depends-гейт.

## Первый запуск в репозитории

```bash
python3 ${CLAUDE_SKILL_DIR}/scripts/task.py init        # создаст tasks/, скопирует CLI в tasks/_cli.py
python3 ${CLAUDE_SKILL_DIR}/scripts/sync_rules.py       # (опционально) сгенерирует AGENTS.md и .cursor/rules для Codex/Cursor
```

(`${CLAUDE_SKILL_DIR}` резолвится в Claude Code автоматически; вне Claude Code подставь путь к папке скилла вручную.)

После `init` проект самодостаточен: все инструменты дальше зовут `python3 tasks/_cli.py <команда>`. Если `tasks/_cli.py` уже есть в репозитории — `init` не нужен.

## Хранилище задач

- Одна задача = один файл `tasks/NNNN-slug.md`. Frontmatter: `id, status, title, files, depends, created`.
- Имя каталога по умолчанию `tasks/`, но его можно переименовать (например, `agent_tasks/`): CLI привязан к каталогу, где лежит `_cli.py`, а не к имени. Ориентир для агентов — каталог с `_cli.py` и `_INDEX.md`; его фактическое имя использовать во всех командах и конвертах вместо `tasks/`. Переименование делать только в паузе: нет задач `in_progress` и открытых worktree, через `git mv` + коммит.
- **Тело файла — спека и единственный источник требований.** Не пересказывай спеку в промптах — передавай путь к файлу. Историю изменений хранит git. Формулировки — императив: «сделай X», не «можно рассмотреть X» — исполнитель буквален, «предложи» даст предложение, а не изменение.
- Статусы: `open → in_progress → done`, плюс `blocked`. Менять статусы **только через CLI** — иначе разъедется индекс.
- `files` во frontmatter — честный список затрагиваемых путей: от него зависит планирование параллельности. Уточняй его при изменении скоупа (правкой frontmatter + `python3 tasks/_cli.py index`).
- `tasks/_INDEX.md` — генерируемый обзор (таблица id/статус/файлы/зависимости). Читай его для ориентации вместо сканирования всех файлов. **Руками не редактировать**; после любых сомнений — перегенерировать.
- Закрытые задачи со временем сжимаются: `archive N --summary "..."` переносит done-задачу в `tasks/archive/` целиком, а в `_ARCHIVE.md` остаётся выжимка 2–3 строки (итог приёмки + spent). `_INDEX.md` показывает живые задачи и одну агрегатную строку про архив — стоимость ориентации не растёт с историей проекта. Архив read-only: статусы архивных задач не меняются, ID не переиспользуются; `depends` на архивную задачу считается выполненным. Триггер — вручную (CLI подскажет при ~50 done); git-история хранит полные файлы в любом случае.
- Файлы с префиксом `_` задачами не являются.
- Все мутации `tasks/` делает только оркестратор. Исполнители каталог `tasks/` не трогают вообще.

## Мини-CLI

Все команды: `python3 tasks/_cli.py <cmd>` из корня репозитория.

| Команда | Что делает |
|---|---|
| `new "Заголовок" --files a.py,b.py --depends 0003 [--risk 0..3]` | создать задачу с шаблоном спеки; `--risk` пишет `risk:` во frontmatter (без флага поле не пишется) |
| `list [--status open]` | таблица задач |
| `view N` | показать спеку задачи |
| `ready` | что можно диспатчить: deps выполнены, файлы не заняты in_progress; предупреждает о пересечениях между готовыми и о цикле deps; задачи с `risk>=2` помечены `[rN]` |
| `ready --waves` | топосорт open-задач по волнам параллельного диспатча (внутри волны файлы не пересекаются); детект циклов зависимостей → `⚠ ЦИКЛ`, exit 1 |
| `start N` | взять в работу |
| `block N --reason "..."` / `unblock N` | заблокировать / снять блокировку (block пишет строку в `## Runs` + секцию `## Блокировка`). Lifecycle enforced: `block` только из `open`/`in_progress`, `unblock` только из `blocked`, иначе exit 1 (обход — `--force` с пометкой в `## Runs`); в multi подчиняются owner-guard, как `close`/`return` |
| `return N --reason "..." [--owner X] [--spent "..."]` | вернуть задачу исполнителю на доработку: статус остаётся `in_progress`, инкремент `returns:`, запись в `## Возврат N` и `## Runs`; 3-й возврат → напоминание circuit breaker |
| `verify N [--base main] [--allow tests/]` | сверить дифф ветки `task/NNNN` с заявленными `files`: ловит правки вне скоупа и любые прикосновения к `tasks/`; exit 1 при нарушении |
|  `close N --note "..." [--spent "sonnet(2):34k,serena(2):2k"]` | закрыть с записью приёмки (+ строка в `## Runs`); `--spent` — фактический расход токенов по моделям. Lifecycle enforced: `close` только из `in_progress` (сначала `start N`), иначе exit 1; обход — `--force` (пишет предупреждение и след в `## Runs`) |
| `archive N --summary "..."` / `archive --done` | сжать done-задачу: файл → `archive/`, выжимка → `_ARCHIVE.md`; без `--summary` возьмёт первое предложение приёмки |
|  `stats [--init "..."]` | сводка: задачи по статусам, расход по моделям, среднее на закрытую задачу (числитель и знаменатель — одно множество done со `spent`, «включая архив»; переоткрытые/не-done в среднее не входят — отдельная строка «учтено вне закрытых»), во сколько раз индекс дешевле чтения всех спек. В multi синхронизируется перед выводом (как `list`/`view`/`ready`) |
| `calibrate [--set "sonnet:12k,serena:8k"]` | показать/сохранить цены спавна в `_CONSTANTS.md`; stats подхватывает их автоматически |
| `index` | перегенерировать `_INDEX.md` |
| `lock --owner <метка> [--force]` | захватить трекер на сессию оркестратора (файл `_LOCK`); без `--owner` — показать статус; exit 1, если залочен другим. Замок **enforced**: под чужим `_LOCK` мутирующие команды отвергаются (exit 1), а не просто предупреждают. Владелец сессии для мутаций — `--owner` команды → env `TASK_OWNER` → нет; несовпадение → отказ, осознанный обход — `--force`. |
| `unlock --owner <метка> [--force]` | снять lock; чужой — только `--force` с подтверждения пользователя |
| `multi on/off/status`, `sync`, `merge-main N`, `start N --owner X` | мультиоркестраторный режим (несколько оркестраторов на одном репо) — протокол в скилле `/task-multi`; в одиночном режиме не нужны. В multi `merge-main` синхронизирует трекер (`sync`) до depends-гейта — гейт судит по свежему `task-sync`, а не по устаревшей локальной копии; `multi status` тоже синхронизируется перед выводом |
| `return N`, `ready --waves` | работают в ОБОИХ режимах (single и multi); в multi `return` подчиняется owner-guard, как `close`/`block` |

Каждая мутирующая команда сама перегенерирует индекс.

## Конвейер

0. **Захват трекера (предохранитель от второго оркестратора).** Перед первой операцией с задачами в сессии: `python3 tasks/_cli.py lock --owner <метка>` (метка сессии, напр. `cc-0704-1512` — инструмент+дата+время). Exit 1 = трекер уже держит другой оркестратор: **не работать с `tasks/` вообще**, сообщить пользователю owner и возраст lock. Перехват `--force` — только после явного подтверждения пользователя, что та сессия мертва (ориентир: возраст >24ч помечается как протухший). По завершении работы конвейера в сессии — `unlock --owner <метка>`. Замок **enforced**: под чужим `_LOCK` любая мутирующая команда (`new`/`start`/`block`/`unblock`/`close`/`return`/`archive`/`calibrate --set`) отвергается с exit 1 — свою сессию задавай `--owner <метка>` на команде или через env `TASK_OWNER=<метка>` (единожды на сессию), тогда мутации проходят; read-команды (`list`/`view`/`ready`/`stats`) под чужим замком работают, печатая напоминание. `_LOCK` — runtime-состояние, в git не коммитить (добавь в `.gitignore`).
1. **Декомпозиция.** Разбей работу на компактные, по возможности независимые задачи. Для каждой: `new`, затем заполни спеку в теле файла (Задача / Контекст / Критерии приёмки / Вне скоупа). Критерии — проверяемые, без «сделать хорошо». Если подключён Codebase-Memory — прогони impact analysis, чтобы список `files` был полным, а не угаданным.

   **Правило проводки.** Задача, добавляющая новый API/фичу, обязана включать точку интеграции (потребителя: main loop, init-путь, вызывающий модуль) в `files` и критерий приёмки «фича наблюдаема через реальный boot/e2e-путь» (живой вызов, ненулевой счётчик), а не только unit-тесты. Модуль без проводки = мёртвый код в прошивке; «подключим потом» без отдельной задачи в трекере — не принимается.

   **Правило общих ресурсов.** Аллокации из общих пространств (namespace, ключевые диапазоны, адреса регистров, ID) приколачивай в спеке конкретными значениями при декомпозиции — не оставляй выбор исполнителю: `ready` ловит пересечения по файлам, но не по логическим ресурсам. При параллельной волне — сводная карта аллокаций волны в спеках всех затронутых задач.

   **Правило симметричных кейсов.** Если спека флагует опасность (гонку, edge case) для одного кейса механизма — явно перечисли все симметричные кейсы того же механизма (например, «reg 1000 И reg 1001», а не только тот, где опасность найдена): исполнитель чинит букву спеки и зеркальный кейс пропустит. Ревьюеру в задание — adversarial-проверка соседних кейсов относительно каждого найденного/исправленного бага.

   **Правило зачистки при удалении.** Задача, удаляющая или переименовывающая компонент, включает в `files` документацию и комментарии (README, заголовки, соседние модули) и критерий «grep по всему репо»: present-tense упоминания удалённого компонента = возврат; исторические «formerly/legacy/dropped»-упоминания корректны, их не трогать.

   **Правило стейл-комментов при смене поведения.** То же для изменения поведения живого компонента: комменты и доки, описывающие СТАРОЕ поведение затронутых путей, после изменения — ложь (класс двух REJECT: «never writes flash» при BL, начавшем писать). Задача, меняющая наблюдаемое поведение, получает критерий «grep по формулировкам старого поведения в затронутых файлах и доках»; исполнителю этот же grep входит в самоопровержение перед отчётом (п.6 конверта).

   **Правило гипотез.** Непроверенный контекст в спеке помечай гипотезой («предположительно X — проверь перед использованием»), а не фактом: исполнитель, получивший неверную гипотезу как факт, построит на ней, вместо того чтобы усомниться. Факт в спеке = проверено оркестратором или читающим субагентом (ссылка файл:строка, прогон); всё остальное — с пометкой.

   **Правило первоисточника (синтез-задачи).** Задача, чей выход — синтез
   источников (гайд, сводка, wiki-страница, выжимка, «извлечённые правила»),
   обязана указывать в спеке самый глубокий доступный источник (транскрипт,
   не пересказ; оригинал, не парафраз) и критерий «каждое утверждение несёт
   указатель (файл:строка / URL / таймкод), сверенный с этим источником».
   Совпадение двух производных одного источника — не подтверждение.

   **Граница суждения.** Читающие/разведывательные субагенты — глаза, не
   голова: find / list / measure / quote / count / grep / прогон. Не их:
   choose / recommend / rank / decide. Субагент вернул рекомендацию →
   это сырьё: решение перевыведи сам, в спеку не копируй без собственного
   обоснования. Нужен выбор из N вариантов → субагент возвращает все N
   с объективными атрибутами, выбирает оркестратор.

   **Правило risk-score.** При `new` проставляй `--risk`: **0** = тривиальное (docs/тесты/конфиг без логики), **1** = локальная логика (один модуль, изолированный эффект), **2** = общие модули/данные (несколько потребителей, разделяемое состояние), **3** = flash/загрузчик/протокол/безопасность. Маппинг на приёмку: risk 0 — `verify` без ревьюера; risk 1 — ревью диффа читающим субагентом; risk 2 — полный конверт ревьюера-опровергателя; risk 3 — ревьюер + обязательный HW/e2e-прогон. `ready`/`ready --waves` помечают задачи с `risk>=2` суффиксом `[rN]`, чтобы приёмочная нагрузка волны была видна заранее.
2. **Планирование.** `ready` показывает, что можно запускать. Задачи с пересекающимися `files` — строго последовательно; непересекающиеся — можно параллельно (в отдельных worktree). Спорные развилки фиксируй в спеке вопросом к пользователю до диспатча.

   **Режим исполнения.** Приоритет: (1) явное указание пользователя — в вызове скилла или по ходу сессии («работай параллельно», «дальше последовательно») — действует до отмены; (2) иначе решает оркестратор: по умолчанию последовательно, параллель — когда задачи долгие и `ready` подтверждает совместимость. Механика следует из режима автоматически: параллель → worktree на каждую задачу; последовательно → основное дерево, ветка `task/NNNN`. Директива «параллельно» — это предпочтение, а не отмена безопасности: пересекающиеся по `files` и связанные `depends` задачи идут последовательно в любом случае — параллелится только то, что `ready` показывает совместимым, остальное честно сообщается пользователю.

   **Волновой план-стоп.** Перед диспатчем — таблица волны в чат
   (задача / модель / depends / parallel / worktree) и явный апрув
   пользователя. Исключения: пользователь явно делегировал выбор
   («решай сам», «дальше без согласований» — действует до отмены) —
   оркестратор решает и диспатчит, таблицу всё равно показывает;
   autonomous-режим — таблица в лог без ожидания. Частичный апрув
   («да, но…») — не апрув: разреши «но» до диспатча.
3. **Диспатч указателем.** Перед запуском исполнителя: `start N`, затем **закоммить `tasks/` в текущую ветку** — worktree ответвляется от HEAD, и незакоммиченные спеки исполнитель в своём дереве не увидит. В промпт исполнителю — конверт (ниже) с путём к файлу задачи, а не текст спеки.

   В конверт подставляй {reports_dir} — каталог отчётов вне git
   (в Claude Code: `<repo>/.reports/`, в .gitignore). Отчёты-файлы —
   страховка от гибели агента и сессии: дайджест потерян → отчёт цел.

   Спеки, написанные заранее пачкой (нормально для волны): перед диспатчем
   каждой — однострочная сверка «дифф уже принятых задач волны не сломал
   допущения этой спеки?» (якоря, номера строк, имена символов, счётчики).
   Сломал — поправь спеку до диспатча, не заставляй исполнителя спотыкаться
   о блокер-расхождение.
4. **Исполнение.** Исполнитель работает строго в рамках конверта. При блокере — останавливается и отчитывается; оркестратор делает `block N --reason`.
5. **Приёмка.** Сначала `verify N` — детерминированный контроль скоупа (дифф против `files`, прикосновения к `tasks/`); при провале — либо возврат исполнителю, либо осознанное обновление `files` в спеке (+ `index`). Затем оркестратор проверяет дифф против критериев приёмки (это можно поручить читающему субагенту). **Ссылкам исполнителя не верить на слово:** каждый закрытый критерий подтверждается конкретной ссылкой файл:тест, чьё существование проверено (grep по имени в диффе/дереве), а прохождение — фактическим запуском; заявленный, но не найденный тест — фабрикация и автоматический возврат задачи. Успех: `close N --note "..."`, слить ветку, затем закоммитить `tasks/` (файл задачи со статусом и приёмкой + индекс) с сообщением `task NNNN: приёмка` — каждый цикл задачи завершается коммитом, история `tasks/` в git и есть журнал конвейера. Провал — лестница эскалации; каждая итерация фиксируется `return N
   --reason "..."` (инкремент `returns:`, запись в `## Возврат N` и `## Runs`,
   статус остаётся `in_progress`; circuit breaker трипается на третьем возврате):
   1-й фейл → тому же исполнителю, в его живой контекст, ДОСЛОВНЫЙ список
      находок (файл:строка, ожидалось/получено — не «тесты упали, разберись»);
      у автора контекст в голове, диагноз ему не нужен — только координаты.
   2-й фейл → переспавн исполнителем ступенью выше (модель/effort по таблице
      «Модели субагентов»). «Тот же агент с бóльшим effort» невозможен — effort
      фиксируется при спавне; контекст восстанавливается из артефактов: спека +
      ветка task/NNNN + отчёт предшественника ({reports_dir}) + полный диагноз
      ревьюера. Стейл-контекст первого исполнителя — частая причина зацикливания,
      свежий взгляд на те же артефакты часто разблокирует сам по себе.
   3-й фейл → `block N --reason` + короткий диагноз пользователю: что пробовали,
      где падает, гипотеза почему. Конвейер продолжается на независимых задачах.

   **Правило circuit breaker.** 3-й возврат одной задачи = стоп-сигнал, не повод для 4-й итерации: не возвращай снова, а эскалируй пользователю. Варианты эскалации: переспецификация задачи (спека неполна/противоречива), смена исполнителя или модели, разбиение задачи на более мелкие. CLI напоминает об этом при `returns>=3` (`⚠ ЦИКЛ: 3 возврата`). Сам счётчик всё равно веди через `return` — но следующий шаг после 3-го это разговор с пользователем, а не диспатч.

   **Правило дедупа фидбека.** Перед повторным возвратом сравни новый провал с текстом последнего `## Возврат N`: если симптом тот же и дифф исполнителя не изменился — это не новая итерация, а зацикливание (исполнитель не сдвинулся). Не повторяй тот же возврат по кругу — эскалируй, как в circuit breaker. Счётчик `returns` инкрементируй через `return` в любом случае (честный журнал попыток), но не жди, что 4-й одинаковый возврат сработает.

   **Vacuous-pass контроль.** Существующий и зелёный тест — ещё не доказательство: новый регрессионный/guard-тест обязан уметь падать. Исполнитель показывает его провал без фикса (прогон на базовой ветке или с временно отключённым фиксом) — тест, который не может упасть, проверяет пустоту. Находка класса «тесты зелёные, но защита молча умерла» (ослабленная ради прохода проверка, заглушка в никуда, ассерт мимо изменённого пути) — безусловный возврат, даже если все критерии формально закрыты.

   **Ревью merge-коммитов оркестратора.** Скоуп приёмки включает и коммиты самого оркестратора — merge веток, конфликт-резолвы, fixup-и: место слияния — где тихая поломка прячется чаще всего, и ни один исполнитель слитый результат не видел. Их проверяет читающий субагент, не автор.

   **Долг из ревью и отчётов.** Не-блок находки не умирают в приёмочной записи: материальная находка (пропущенный тест, запаркованный дефект, отложенная зачистка) — от ревьюера ИЛИ из секции «Смежные находки» отчёта исполнителя — сразу `new` (debt-задача, `depends` по необходимости) + номер этой задачи ссылкой из приёмки. Нематериальное — явная пометка «задачи не требует». «Подключим/допишем потом» без задачи в трекере — не существует. Debt-задачи создавать с `--debt-from NNNN` (номер задачи-источника): пишет `debt_from:` во frontmatter, `doctor` считает по родителям и предупреждает от 4-й подряд.

   **Учёт токенов (опционально).** При закрытии: `--spent "sonnet(2):34k, haiku(1):6.2k, serena(2):2k"` — в скобках число спавнов модели (для MCP-ключей вроде serena — число спавнов с подключённым инструментом; без скобок считается 1). Цифры — только из счётчиков инструмента (результат Task-вызова, `/cost`); если счётчика нет, поле пропускается, не выдумывается. `stats` агрегирует расход и при наличии калибровки автоматически раскладывает его на работу и инициализацию. Это учёт потраченного, а не «сэкономленного».

   **Калибровка (`/task calibrate`).** По запросу «откалибруй цены спавна» оркестратор: (1) спавнит тривиального субагента («ответь одним словом OK») на каждую используемую конфигурацию — читатель haiku без MCP, кодер sonnet без MCP, кодер sonnet с Serena, при необходимости opus и codebase_memory; (2) берёт input-токены каждого из счётчиков Task-результатов; (3) ключи моделей = базовый спавн, ключи инструментов = разность (с инструментом − без); (4) сохраняет: `python3 tasks/_cli.py calibrate --set "sonnet:12k,haiku:4k,serena:8k"`. Константы лежат в `_CONSTANTS.md` рядом с задачами (не в папке скилла: CLI после init не знает её путь, у Codex/Cursor её нет, а состав инструментов — свойство проекта) и версионируются git. Разложение: работа = расход − спавны × константа (модельно-инвариантная часть — та же спека и файлы стоили бы примерно столько же другой модели); оценка не учитывает кэширование префикса и разное число итераций у моделей. Доля инициализации стабильно выше ~30% — задачи слишком мелкие, укрупняй декомпозицию. Перекалибровка — при смене версии инструментов, состава MCP или системного промпта; `stats --init "..."` разово переопределяет константы без записи.

Защита от коллизий двух агентов на одном файле — три слоя: `ready` не даёт диспатчить пересекающиеся по `files` задачи (предотвращение); параллельные исполнители сидят в разных worktree, физически один файл одновременно не редактируется, коллизия откладывается до мерджа (изоляция); `verify` + merge-конфликт на приёмке ловят выход за скоуп и недекларированные пересечения (обнаружение). Слабое звено — полнота `files`, поэтому заполняй его через impact analysis, а не наугад.

## MCP-инструменты (если подключены)

Если в сессии доступны Serena и/или Codebase-Memory — используй их по разделению ролей ниже. Если не подключены — работай обычными средствами, конвейер от них не зависит.

Use Serena for:
- symbol-level edits
- refactoring
- rename/move operations
- reading exact symbol bodies
- finding definitions/references through LSP
- applying changes safely

Use Codebase-Memory for:
- call graph questions
- impact analysis
- architectural overview
- dependency chains
- "where does this flow start/end?"
- cross-module relationships
- persistent codebase memory across sessions

**Do not query both tools for the same lookup** unless the first result is incomplete or contradictory.

**Пропорциональность, не ритуал.** Инструменты Serena стоят контекста при каждом спавне субагента (загрузка определений + активация проекта), поэтому: точечные правки в маленьких файлах — обычным edit; Serena подключай, когда symbol-level реально выигрывает — большие файлы (чтение тела символа вместо всего файла), ренеймы/перемещения с множеством references, изменение сигнатур, незнакомый код. Оркестратор может указать это прямо в спеке («правки через Serena» / «Serena не нужна») — тогда исполнитель не решает сам. Решение фиксируй с причиной («Serena не нужна: файлы маленькие, правки точечные» / «правки через Serena: ренейм с 40+ references»), а факт — в приёмочной записи: накопленные причины — обратная связь для калибровки самого этого правила (где отказ от Serena стоил итерации, а где подключение не окупилось).

Привязка к этапам конвейера:
- **Декомпозиция (оркестратор / читающий субагент):** Codebase-Memory — impact analysis и dependency chains, чтобы честно заполнить `files` во frontmatter и выставить `depends`. Точность `files` = качество планирования параллельности.
- **Исполнение (кодящий субагент):** Serena — правки на уровне символов вместо сырых патчей; definitions/references перед изменением сигнатур. При работе в worktree Serena активируется на пути этого worktree, не на основном дереве.
- **Параллельный режим — Serena максимум у одного исполнителя одновременно.** MCP-серверы общие на сессию: Serena-сервер один, `activate_project` — его глобальное состояние, два параллельных исполнителя перещёлкивают проект друг у друга и символьные операции уходят не в то дерево. Оркестратор при параллельном диспатче даёт Serena одной задаче, остальным пишет в спеку «Serena не нужна», либо ставит Serena-тяжёлые задачи (ренеймы, смена сигнатур) последовательно.
- **Приёмка (читающий субагент):** Serena find_references — проверить, что изменённые символы не сломали вызовы за пределами диффа; Codebase-Memory — если критерий приёмки затрагивает архитектуру.

## Конверт исполнителя (шаблон промпта)

```
Ты — исполнитель задачи {ID}.
1. Прочитай спеку: {путь к tasks/NNNN-*.md}. Спека — единственный источник требований.
2. Рабочее место: {worktree/ветка}. За его пределы не выходи.
3. Запрещено: править что-либо в tasks/ (включая свою спеку и _INDEX.md);
   мержить в main; трогать файлы вне секции «Контекст»; расширять скоуп;
   менять зависимости проекта, если этого нет в спеке явно;
   спавнить собственных субагентов (Agent/Task tool) — всю работу делаешь
   сам; нужна помощь — это блокер, остановись и отчитайся (п.5).
4. Коммиты с префиксом "task {ID}: ...". Вся работа должна быть закоммичена
   до финального отчёта: незакоммиченные изменения не видны ни verify, ни merge —
   их не существует для приёмки. Рабочее дерево перед отчётом — чистое
   (кроме игнорируемых артефактов).
5. Блокер — остановись и опиши причину. Не обходи его самостоятельно.
   Расхождение «факта» из спеки с живым кодом — тоже блокер: не строй
   на догадке и не правь спеку — остановись, отчитайся с файл:строка.
6. Перед финальным отчётом — самоопровержение: пройди критерии приёмки как
   противник (чем моё изменение ломается? какой соседний кейс я не тронул?);
   прогони grep по формулировкам СТАРОГО поведения в затронутых файлах —
   комменты/доки, описывающие поведение до изменения, почини до отчёта;
   что опровергнуть не смог, но и подтвердить нечем — честно в отчёт.
7. Перед финальным сообщением запиши ПОЛНЫЙ отчёт в файл
   {reports_dir}/{ID}-coder.md (путь даёт оркестратор; полнота — file:line,
   вывод прогонов, все детали). Финальное сообщение — самодостаточный дайджест
   ≤15 строк + путь к файлу: каждый факт, нужный оркестратору для решения, —
   инлайн в дайджесте, не «см. файл». Дальше в дайджесте: что сделано; какие
   критерии приёмки закрыты (по пунктам); что не удалось и почему; смежные
   проблемы вне скоупа — отдельной секцией «Смежные находки (не чинил)»:
   чинить их запрещено п.3, молчать — тоже (оркестратор превратит материальное
   в debt-задачу); использовалась ли Serena и почему (одной строкой:
   «Serena: не использовалась — <причина>» / «использовалась для <что>»).
8. Если подключены Serena / Codebase-Memory — следуй правилам раздела
   «MCP-инструменты»: правки через Serena, анализ связей через Codebase-Memory,
   не дублируй один и тот же lookup в оба инструмента.
   ВАЖНО: перед первой правкой активируй Serena на пути СВОЕГО worktree
   ({worktree}), а не основного дерева репозитория — иначе правки уйдут не туда.
```

## Конверт ревьюера (шаблон промпта)

Ревью — опровержение, не пересказ: сформулируй изменение как претензию и поручи её атаковать. Ревьюер read-only, код не правит.

```
Ты — ревьюер задачи {ID}, read-only: ничего не правишь.
Субагентов не спавнишь — проверки гоняешь сам.
Претензия: {точная формулировка того, что утверждается — «коммиты {X..Y}
чинят {Z}, контракт {W} сохранён, тесты зелёные»}.
Твоя работа — ОПРОВЕРГНУТЬ претензию. Не суммаризируй дифф и не оценивай
стиль — ищи свидетельства ложности: читай фактический дифф, гоняй проверки
(тесты, grep, сборку) сам.
Охоться по категориям (подстроены под тип изменения):
{рефактор/перенос: дрейф тела при переносе; направление зависимостей;
 потерянные экспорты/точки входа; моки и тесты, после переноса проверяющие
 пустоту}
{багфикс: гонки вне happy-path; краевые (границы, пустое, переполнение,
 wrap); проглоченные ошибки; регресс для зависевших от старого поведения;
 симметричные кейсы того же механизма}
{синтез/доки: слова-связки, добавленные при сжатии («всегда», «когда»,
 «после», «большинство», «следовательно»), сверить с первоисточником —
 галлюцинации живут в связках, не в цитатах; указатели бьются по
 первоисточнику, а не по другой производной; факт без указателя}
Всегда, независимо от типа:
- прогоняет ли тест-сьют реально изменённый путь, или проходит вакуумно
  (ослабленный ассерт, заглушка мимо кода, тест не может упасть);
- остались ли в затронутых файлах/доках комменты и формулировки, описывающие
  поведение ДО изменения (present-tense ложь; исторические
  «formerly/legacy»-пометки корректны);
- что из отчёта исполнителя ты НЕ можешь независимо подтвердить по диффу
  и собственному прогону.
НЕ находка (не флагай и не основывай на этом REJECT):
- пред-существующее: дефект есть и на базовой ветке, не внесён коммитами
  {X..Y} — максимум упомяни отдельной строкой «pre-existing, вне вердикта»;
- ловится компилятором/линтером/сборкой (импорты, типы, формат) — сборка
  и тесты гоняются отдельно, ты проверяешь то, что они не ловят;
- намеренное подавление на месте (lint-ignore и т.п.), соответствующее спеке;
- изменение поведения, явно требуемое спекой — это ТЗ, а не баг.
Материальная находка = регресс/поломка, ВНЕСЁННАЯ этими коммитами в
заявленном скоупе задачи.
Полный протокол ревью (категория→свидетельства) запиши в
{reports_dir}/{ID}-review.md; финальное сообщение — вердикты + находки.
Вердикт по каждой категории: CONFIRMED (файл:строка) | NO FINDINGS.
Каждая CONFIRMED-находка обязана нести строку сверки со списком «НЕ находка»:
«FP-check: не подпадает — <почему не пункты 1–4>». Находка без FP-check
в итог не входит и REJECT не обосновывает.
Итог: ACCEPT | REJECT (находки в порядке приоритета). REJECT при любой
подтверждённой материальной находке.
```

Итог арбитрирует оркестратор: сперва прогони каждую находку через FP-список сам — REJECT, чья единственная находка подпадает под «НЕ находка» или лишена строки FP-check, отклоняется без возврата исполнителю (инцидент 0035); затем: находка → фикс той же веткой (в идеале — тем же исполнителем, у него контекст); повторные фейлы — по лестнице эскалации (п.5 Конвейера); «формально верно, но так требовала спека» → отклонить с обоснованием в приёмке; материальное не-блок → debt-задача (см. «Долг из ревью и отчётов»).

## Режим без субагентов

Если субагентов нет (Codex, Cursor, одиночная сессия): роли оркестратора и исполнителя исполняются последовательно в одной сессии либо в отдельных сессиях/вкладках — по одной задаче за раз. Конверт и запреты действуют без изменений: взяв роль исполнителя, не трогай `tasks/`; вернувшись в роль оркестратора — проведи приёмку и только тогда меняй статусы.

## Claude Code / Desktop: механика диспатча

- Исполнители — субагенты через Task tool: читающие на haiku, кодящие на sonnet (или модели из переопределения при вызове).
- Правило выбора: **параллельные исполнители — обязательно в отдельных worktree, даже если их всего два** (ветки изолируют историю, но не рабочий каталог: два агента в одном дереве смешивают правки, гонятся за git-индекс и ломают verify). **Последовательные исполнители worktree не требуют** — каждый работает в основном дереве на своей ветке `task/NNNN`, по одному за раз. Ветка на задачу обязательна в любом режиме: без неё нет `verify` и отката провальной работы.
- Создание worktree для параллельной задачи:
  ```bash
  git worktree add ../<repo>-wt-NNNN -b task/NNNN
  ```
- После приёмки: слить `task/NNNN`, затем `git worktree remove --force ../<repo>-wt-NNNN` — обычный `remove` упадёт на артефактах тестов/сборки (`__pycache__` и т.п.), которые исполнитель оставил в дереве. Убедись, что `.gitignore` покрывает такие артефакты, иначе они попадут в коммиты исполнителей.
- Проверку критериев приёмки по диффу поручай читающему субагенту (haiku): вход — путь к спеке + дифф, выход — вердикт по каждому критерию.
- **В каждый промпт субагенту — явный запрет спавна собственных субагентов.** Вложенный фоновый спавн = потерянная работа: субагент, запустивший фоновых детей, завершает ход «жду уведомлений» и умирает вместе с ними при закрытии сессии, не вернув результата. Всю работу субагент делает сам; нужна помощь — останавливается и отчитывается оркестратору.
- Восстановление после гибели агента — сперва `SendMessage` живому агенту, затем `.reports/`, затем респавн по рецепту `RECOVERY.md#agent-death`.

## Кросс-форматная поставка

Канонический текст — этот SKILL.md. SKILL.md — открытый стандарт (agentskills.io), и Codex CLI читает его нативно: та же папка скилла кладётся в `<repo>/.agents/skills/task/`, вызов — `$task` или автотриггер по description. Конвертация для Codex не нужна.

`scripts/sync_rules.py` — запасной путь: раскладывает инструментонезависимое ядро (между маркерами `core:start/core:end`) в `AGENTS.md` (всегда загружаемый контекст Codex — в отличие от скилла, который подгружается по совпадению) и `.cursor/rules/task.mdc` (Cursor). После правок ядра перезапусти `sync_rules.py` в целевом репозитории — блок в AGENTS.md обновится по маркерам, остальное содержимое файла не пострадает.

Маркеры `<!-- core:start -->`/`<!-- core:end -->` в этом файле — не оформительский комментарий, а **контракт с `sync_rules.py`**: он ищет их по имени и синхронизирует ровно то, что между ними (плюс второй такой же блок в `../task-multi/SKILL.md` — Ф11, задача 0064); текст вне маркеров в AGENTS.md/`.cursor` не попадает. Переименовывать или убирать маркеры без правки `sync_rules.py` нельзя — экспорт молча перестанет находить блок.

Почему так спроектировано (не split SKILL.md, файл-на-задачу, три слоя защиты и т.д.) — `DESIGN.md`.
