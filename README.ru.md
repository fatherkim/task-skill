# task — конвейер «оркестратор + руки-агенты»

Skill для Claude Code: декомпозиция работы на задачи в локальном файловом
трекере (`agent_tasks/*.md` + мини-CLI на stdlib-Python) и раздача их кодящим
субагентам. Без GitHub Issues, без внешних зависимостей.

*English version: [README.md](README.md).*

## Что внутри

```
.claude/skills/task/
├── SKILL.md              # сам скилл (канонический текст)
├── DESIGN.md             # журнал проектных решений (почему так)
└── scripts/
    ├── task.py           # мини-CLI трекера (new/list/start/close/verify/ready/stats/calibrate)
    └── sync_rules.py     # генерация правил для Codex (AGENTS.md) и Cursor (.cursor/rules)
```

## Ключевые идеи

- **Оркестратор не пишет код.** Он декомпозирует, диспатчит субагентов
  (haiku-читатели, sonnet-кодеры), принимает работу и двигает статусы.
- **Файл-на-задачу** в `agent_tasks/NNNN-slug.md` с frontmatter
  (status/files/depends/spent) — merge-конфликтов между задачами нет.
- **`_INDEX.md` — генерируемый артефакт**: ориентация по индексу в ~40-50 раз
  дешевле чтения всех спек.
- **4 слоя защиты от коллизий**: `ready` (предотвращение), worktree (изоляция),
  `verify` (контроль скоупа диффа), merge-конфликт (детекция).
- **Анти-фабрикация приёмки**: каждый критерий = ссылка file:test, существование
  проверяется grep-ом, прохождение — реальным прогоном.
- **Учёт токенов**: `close N --spent "sonnet(2):141k,opus(1):59k"`, `stats`
  раскладывает расход на работу и инициализацию спавнов (константы —
  `calibrate --set`), доля инициализации >30% ⇒ задачи слишком мелкие.

## Проектные решения (из DESIGN.md)

Почему такая форма, а не очевидные альтернативы:

- **Файл-на-задачу + мини-CLI** — не единый BACKLOG.md, не SQLite, не готовый
  трекер. Один файл = нет конфликтов при параллельной работе; markdown =
  человекочитаемый дифф и спека, правимая текстом; CLI без зависимостей
  работает везде. После `init` CLI копирует себя в трекер-каталог
  (`agent_tasks/_cli.py`) — проект самодостаточен, все инструменты зовут один
  и тот же скрипт.
- **Единственный писатель трекера — оркестратор** (исполнителям запрещено),
  **статусы только через CLI** (`open → in_progress → done`, плюс `blocked`) —
  поэтому генерируемый индекс бесконфликтен и синхронен по построению.
- **`ready` кодирует планирование детерминированно**: зависимости выполнены И
  заявленные `files` не пересекаются с in_progress; предупреждает о
  пересечениях среди готовых задач. Слабое звено — полнота `files`; заполнять
  через impact analysis.
- **Режим исполнения**: директива пользователя («параллельно»/«последовательно»)
  — высший приоритет, дефолт последовательный. Параллельно ⇒ worktree
  обязателен даже для двух агентов — ветки изолируют историю, НЕ рабочий
  каталог (общий каталог = смешанные правки и гонка за git-индекс).
  Последовательно ⇒ основное дерево, ветка `task/NNNN`. Ветка на задачу
  обязательна всегда: на ней держатся `verify` и откат. Директива не отменяет
  безопасность: пересечения по `files` идут последовательно в любом случае.
- **`verify` — детерминированный контроль скоупа** (`git diff main...task/NNNN`
  против заявленных `files`): ловит правки вне спеки и прикосновения к
  трекер-каталогу. Появился потому, что запреты конверта — инструкция, а не
  механизм. Префикс `tests/` разрешён по умолчанию (`--allow` настраивается).
- **Дефолты моделей — в тексте скилла** (haiku чтение / sonnet код),
  переопределение аргументом вызова. Конфиг-файл отвергнут сознательно
  (лишняя сущность).

## Правила, рождённые из инцидентов

Каждое — из реального сбоя в боевом тесте, не из умозрения:

1. **Коммитить трекер-каталог ДО создания worktree** — worktree ответвляется
   от HEAD, незакоммиченная спека исполнителю не видна.
2. **`git worktree remove --force`** — обычный remove падает на артефактах
   тестов (`__pycache__` и т.п.); `.gitignore` обязателен.
3. **Анти-фабрикация в приёмке** — исполнитель дважды отчитывался о
   несуществующих/незапущенных тестах. Отсюда: каждый критерий — ссылка
   file:test, существование проверено grep-ом, прохождение — запуском.
   Заявленный, но не найденный тест = возврат задачи.
4. **Вся работа исполнителя закоммичена до отчёта** — незакоммиченное не
   существует для `verify`/merge. Оркестратор коммитит трекер после каждого
   `close` — его git-история = журнал конвейера.
5. **Символьные тулы (например, Serena MCP) активируются на пути СВОЕГО
   worktree**, не основного дерева.
6. **MCP-тулы — пропорционально, не ритуально**: тулы стоят контекста при
   каждом спавне; мелкие правки — обычным edit. Оркестратор может предписать
   инструменты в спеке.

## Экономика токенов

`--spent "sonnet(2):34k,serena(2):2k"` — скобки = число спавнов (для MCP-ключей
— спавнов с подключённым инструментом, НЕ вызовов: константа = цена загрузки
тулов в контекст). Только цифры из счётчиков инструмента; нет счётчика ⇒ поле
пропускается, не выдумывается. Учитываем потраченное, не «сэкономленное»
(экономия = контрфактуал, измерима только сравнением с baseline).

**Калибровка**: тривиальный субагент на конфигурацию ⇒ его input = константа
спавна; константа MCP = разность (с − без). Константы живут в `_CONSTANTS.md`
рядом с задачами, не в папке скилла (CLI после `init` не знает путь скилла;
у Codex/Cursor его нет; состав инструментов — свойство проекта). `stats`
раскладывает расход: работа (модельно-инвариантна) vs инициализация
(спавны × константа). Доля инициализации стабильно >30% — задачи слишком
мелкие, укрупняй декомпозицию. Оценка не учитывает кэш и разное число
итераций у моделей.

## Установка

Claude Code: скопировать `.claude/skills/task/` в корень своего репозитория —
скилл подхватится как `/task`. При первом использовании CLI бутстрапится в
трекер-каталог (`agent_tasks/_cli.py`).

Codex читает скиллы нативно из `.agents/skills/` — симлинк
`.agents/skills/task → ../../.claude/skills/task` даёт один источник правды.
Запасной путь для Codex/Cursor без поддержки скиллов:

```sh
python3 .claude/skills/task/scripts/sync_rules.py --repo /path/to/repo
```

— извлечёт инструментонезависимое ядро SKILL.md (между маркерами
`core:start`/`core:end`) в `AGENTS.md` и `.cursor/rules/task.mdc`.
В инструментах без субагентов роли оркестратора и исполнителя выполняются
последовательно в одной сессии или в отдельных вкладках; конверт исполнителя
и все запреты действуют без изменений.

## CLI

```sh
python3 agent_tasks/_cli.py new "Заголовок" --files a.c,b.h --depends 0001
python3 agent_tasks/_cli.py list|index
python3 agent_tasks/_cli.py start 0002
python3 agent_tasks/_cli.py verify 0002 [--base main] [--allow tests/]
python3 agent_tasks/_cli.py close 0002 --spent "sonnet(1):69k"
python3 agent_tasks/_cli.py ready          # что можно брать без коллизий
python3 agent_tasks/_cli.py stats          # экономика конвейера
python3 agent_tasks/_cli.py calibrate --set "sonnet:23300,opus:18100"
```

Требования: Python 3 (stdlib), git.

## Замечания и границы

- Трекер-каталог переименовываем (`tasks/` → `agent_tasks/`): CLI привязан к
  своему расположению, `verify` берёт префикс динамически. Переименование —
  только в паузе (нет in_progress/worktree), через `git mv`.
- Нумерация задач без потолка (`\d{4,}`, сортировка числовая).
- Происхождение: порт скилла fable-ruki-agenty с GitHub Issues на локальный
  файловый трекер; требования — без сети/gh, кросс-поддержка Claude Code
  (терминал / VS Code / Desktop), Codex, Cursor.
